Chapter eight. Hooks.

Let's start with what hooks are. A hook is an automation that fires on a specific event: a tool call, a session starting, the context about to be compacted. Unlike skills and commands, you never invoke a hook. The harness triggers it. Hooks are ECC's reflexes: the formatting, gating, secret-scanning, and memory-saving that should happen every time, automatically, without you remembering to do it. In ECC, a configuration file declares the events and matchers, and each entry runs a small Node script. They use Node rather than inline shell, for cross-platform reliability, which was a deliberate hardening introduced in version one-point-eight.

Now let's walk through the event lifecycle. The main events use Claude Code's names, and other harnesses map onto these. There's session start, which fires when a new session begins, and ECC uses it to load previous context, detect your package manager, and inject a memory summary. There's pre-tool-use, which fires before a tool runs, and ECC uses it to remind you about using a terminal multiplexer, to gate a git push, to scan for secrets, to protect config files, and to force a fact-check before the first edit. There's post-tool-use, which fires after a tool finishes, and ECC uses it to format code, type-check, run a quality gate, warn about stray debug logs, and update metrics. There's a post-tool-use-failure event, which captures failure context for diagnostics when a tool errors. There's pre-compact, which fires before context compaction, and saves important state to a file first. And there's stop, which fires when a session ends, writing a session summary, batch-formatting and type-checking, and extracting learnings.

There are several representative ECC hooks you'll see by name. There's a consolidated shell preflight that bundles quality, multiplexer, push, and gate checks. There's a gate-guard fact-force hook that blocks the first edit per file and demands investigation, asking who imports this, what's the schema, and what did the user actually ask, before allowing changes. There's a config-protection hook that blocks edits to linter and formatter configs, steering the agent to fix the code instead of weakening the config. There's a quality-gate hook that runs after edits. There's a console-warning hook for stray debug logs. There's a context-monitor hook that injects warnings about context exhaustion, high cost, scope creep, or tool loops. There's a session-start bootstrap for memory and package-manager detection. And there are the compaction and stop hooks for memory persistence. The actual command strings in the configuration look gnarly, because each one bootstraps the correct plugin root across many possible install locations before delegating to the real script. You never write that by hand. The installer generates it.

Here's one of ECC's nicest features: you can tune hooks without editing any files. You control hook behavior with environment variables. There's a strictness profile, which can be minimal, standard, or strict, defaulting to standard. You can disable specific hooks by their id, in a comma-separated list. You can cap or even turn off the session-start context injection, which is handy for low-context or local models. You can set a session retention window in days. You can keep the context, scope, and loop warnings while hiding the cost estimates. And you can opt in to governance event capture. Each hook is tagged with the profiles it runs under, so the minimal profile quietly drops the noisier ones, while strict turns everything on.

Now, a warning about the duplicate-hooks-file trap. This is the single most-reported hook issue. Claude Code version two-point-one and later auto-loads a plugin's hooks configuration by convention. If you also declare hooks in the plugin manifest, or copy the hooks configuration into your settings file after a plugin install, you get a duplicate-hooks-file error. Here's how to avoid it. Contributors should never add a hooks field to the plugin manifest, and a regression test enforces this. Plugin users should not copy hooks into their settings file, because they're already loaded. And manual users should install hooks through the installer with the hooks-runtime module, which writes them to the right location and leaves the settings file alone.

Let's look at the conceptual shape of a hook. If you peel away the bootstrap wrapper, a hook entry is just a few fields. There's a matcher, which says which tools or events this applies to, like a shell call, an edit, or everything. There's the command, which is the script to run. There are async and timeout settings, so the hook doesn't block the agent and its runtime is capped. And there's an id, which is the handle you use when disabling hooks. There's also an exit-code convention: exit with a failure code only when blocking is intentional, otherwise exit cleanly, and messages should always be actionable. A minimal hand-written example from the guides is a hook that matches edits to JavaScript or TypeScript files, greps for a stray debug log, and prints a reminder to remove it.

Creating hooks the easy way is recommended, because writing hook configuration by hand is fiddly. The shortform guide recommends the hookify approach: run the hookify command and describe what you want conversationally, and it generates the hook for you. ECC also ships companion commands to list, configure, and get help with hookify. If you do author one manually, validate it with the provided script.

So what do hooks give you in practice? With hooks on the standard profile, a normal editing session quietly gets: auto-formatting and type-checking after edits, so no more forgetting to run the formatter; secret detection before risky tool calls and before prompts are submitted; a git-push review gate so pushes aren't silent; a reminder to use a terminal multiplexer for long-running commands; memory persistence across sessions; continuous-learning capture feeding instincts; and a context monitor that warns you before you hit a wall. That's a lot of senior-engineer discipline for zero ongoing effort.

So, the key takeaways. Hooks are automatic reflexes on tool and session events, mapping a configuration file to Node scripts. The lifecycle runs from session start, to pre-tool-use, to post-tool-use, to pre-compact, to stop. You tune them at runtime with the hook-profile and disabled-hooks environment variables, with no file editing. Never double-load hooks, the duplicate-hooks-file trap, and install them the right way. And use the hookify command to create hooks by description rather than writing configuration by hand.
