Chapter four. Core concepts: the six building blocks.

Everything in ECC is made of six kinds of things. Learn these six, and the whole system snaps into focus. This chapter is the map, and the chapters that follow zoom into each one.

Let's look at the six at a glance. First, agents. These are scoped specialists you delegate to. They live as individual files, and the way you interact is that the orchestrator delegates to them, though you can also request one directly. Second, skills, and these are the star of the show. They're reusable, multi-step workflows bundled with knowledge. Each lives in its own folder. You invoke them, they get auto-suggested, or agents reuse them. Third, commands. These are slash entries that trigger a workflow, and you interact by typing a slash and the command name. Fourth, hooks. These are event-triggered automations, mapped from a configuration file to small scripts, and the way you interact is that you don't. They fire automatically. Fifth, rules. These are always-follow guidelines, both common ones and per-language ones, and they're loaded into context every session. And sixth, MCP configs. These are connectors to external services. You enable them per project, and the model calls the tools they expose. Of all six, skills are the canonical surface. When in doubt, think in skills.

Here's a useful way to hold all six in your head. Rules are the constitution: passive, always present, constraining everything. Skills are the playbooks: the durable how-we-do-X-here. Commands are shortcuts: convenient typed triggers that are increasingly thin wrappers over skills. Agents are the team: specialists the orchestrator hands work to, often using skills. Hooks are the reflexes: they fire on events without anyone asking. And MCP is the phone line: how the model reaches GitHub, a database, or a browser. A single feature request might touch all six. A command kicks off a skill, which delegates to an agent, all constrained by rules, with hooks firing on every edit, and an MCP server fetching data. That choreography is ECC.

Let's zoom in on skills versus commands versus hooks, the three active surfaces. These three are the ones people confuse most, because all three can run a workflow. The difference comes down to the trigger and the durability. A skill is invoked, by you or automatically, and it's the durable unit. New logic belongs here. A command is something you type, and ECC keeps commands for compatibility, but it's migrating their logic into skills. A hook is triggered by an event, like a file edit or a session start, and you never type it. Here's a rule of thumb. Ask yourself: should this run every time X happens, no matter what? If so, it's a hook. Or ask: is this a workflow I choose to run? If so, it's a skill, with an optional command shortcut.

On top of the six blocks, there are two cross-cutting capabilities that tie everything together. They get their own chapters because they're what make ECC feel alive rather than static. The first is memory. Hooks save your session state and reload it next time, so the assistant doesn't start from zero each day. The second is continuous learning. Repeated patterns become instincts, which an evolve command clusters into brand-new skills. The catalog grows from your own usage.

There's also a frontmatter convention worth knowing. Agents and skills are just Markdown files with a small structured header at the top. The header tells the harness how to treat the file. You'll see this pattern constantly: a name, a description of when to use it, and for agents, the tools and model they should use, while skills note their origin. The description field is doing heavy lifting. It's how the harness decides when to surface this agent or skill. Vague descriptions get ignored. Precise ones get used.

So, the key takeaways. ECC is six blocks: agents, skills, commands, hooks, rules, and MCP configs. Skills are canonical. Commands are increasingly thin shortcuts. Hooks are automatic. Rules constrain, agents execute, MCP connects, and memory plus learning make it improve over time. Agents and skills are just Markdown plus a structured header, and the description is what drives auto-selection.
