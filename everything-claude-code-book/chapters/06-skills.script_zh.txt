第六章。技能，Skills。

我们讲一下为什么 skills 是 ECC 的核心。如果 agents 是“谁做事”，那么 skills 就是“事情怎么做”。一个 skill 是一个可重用、自成一体的工作流程：那些步骤、结构、示例、陷阱，有时还有支持文件和 codemaps。一个 codemap 是一份代码库的轻量地图，让助手可以不用烧 token 重新探索就导航到。

ECC 很强调，skills 是标准的工作流程界面。新的工作流程开发应该先落在 skills。Commands 仍然存在是为了兼容，但耐用的逻辑就属于 skills。原因要追溯回之前那套复利工作流程的理念：一个 skill 会被永远重用，跨项目和跨会话，而且随着模型改进，它会变得越来越好。

一个 skill 有三种被使用的方式。第一，你用名字去调用它。第二，当它的描述和你正在做的事相符，它会被自动建议。而第三，一个 agent 会重用它：一个可以执行你部分 skills 的 sub-agent，可以被交一个任务，然后自主地用那些 skills。

现在我们看一下一个 skill 文件的解剖。每一个 skill 住在自己的目录，里面有一个主 skill 文件。它的头部很小。有个名字，和目录一致。有段描述，是驱动自动选取那个“什么时候用”的触发器，所以它要具体。还有一个来源字段，第一方的 skills 会标成 ECC，而引入和贡献的就会标成 community。正文应该包含实用的指引、经过测试的示例，和一个清晰的“什么时候用”部分。一个 skill 文件夹可以装多于只有主文件，包括额外的参考文件、模板，和 codemaps。

我们剖析一个真正的 skill：测试驱动开发工作流程。它是 ECC 里面最重要的其中一个，也是写你自己那个的好模板。它的结构是这样。第一，什么时候启动：新功能、修 bug、重构、新的端点或者组件。第二，核心原则：代码之前先写测试、八成以上的覆盖率，和三种测试类型，也就是单元、集成和端到端。第三，git 检查点：每个阶段之后 commit，在消息里附上证据，在当前的分支上。第四，七个步骤：写用户旅程、生成测试用例、执行测试并确认它们失败、实现最小的代码、执行测试并确认它们通过、重构，和验证覆盖率。第五，红色关卡：一个关于什么才算是一个有效的失败测试的严谨定义，无论是运行时失败还是编译时失败，这样你就不能造假。正如这个 skill 所说，一个只是写了、但没有编译和执行过的测试，不算是红色。第六，测试模式：给单元测试框架、API 集成，和浏览器测试的具体示例。而第七，模拟示例、覆盖率门槛、要避免的常见错误，和成功指标。留意一下这个 skill 有多少是关于严谨的。它不是只说“写测试”。它精确地定义了什么是一个有效的失败测试，并且拒绝让你跳过红色那一步。那种精准，就是让一个 skill 可信到足以委派给一个 agent 的原因。

现在我们游览一下这个 skill 目录，不是一份完整清单，而是感受一下那些类别。有超过 260 个 skills，所以背它们是没意义的，但知道那些群组就很强大。在工程核心，你会找到关于测试驱动开发、验证循环、评估框架、安全审查和扫描、编程标准、API 设计、后端和前端模式、数据库迁移、部署和 Docker 模式、端到端测试、错误处理，和 git 工作流程的 skills。有语言和框架包，每个都为一个指定的技术栈捆绑了模式、安全、测试驱动开发和验证，涵盖 Python、Go、Rust、Java、Kotlin、Swift、C++、Perl、Django、Laravel、NestJS、React、Next.js、Bun、Flutter 和 Android。有一个关于上下文和成本工程的群组，有关于策略性压缩、上下文预算、token 预算建议、迭代式检索、成本感知流水线、内容哈希缓存，和在处理结构化文本时挑正则表达式还是 AI 模型的 skills。有一个学习和元层级的群组，有持续学习、skill 盘点、rules 提炼、agent 评估、框架构建、代码库上手，和配置 ECC 本身。有一个编排和循环的群组，有自主循环、持续代理循环、多路复用工作流程，和计划编排。而还有一个研究、内容和操作者的群组，有深度研究、搜索优先、市场研究、文章写作、内容引擎、跨平台转发、投资者材料、演示文稿制作，和视频剪辑，再加医疗、物流、海关、能源和制造的领域包，和一个预测市场包。这次游览的重点是：无论你在做什么，很可能都有一个 skill 合用。用 consult command 去找它。

快速讲一讲 skills 存在哪，也就是放置策略。ECC 区分精选的 skills，它们住在 repo，和生成或者引入的 skills，它们住在你的主目录配置下面。策略是这样。Claude Code 只从你 skills 目录的直接子项目加载 skills，所以手动安装不要嵌套放在一个子文件夹里面。给做手动安装的新用户：复制核心和通用的 skills，加上搜索优先那个 skill，只在需要时才加冷门的包。而生成的 skills，也就是通过 skill-create command 创建的，连同学到的 skills，会落在你的代理数据主目录下面。如果你想要完整的规则，repo 里面有关于放置和 skill 开发的权威文档。

你怎样创建 skills 呢？有两条路。手动那条：创建一个带主 skill 文件的目录，给它一个带名字、描述和来源的头部，然后写一个带“什么时候用”部分、步骤和经过测试示例的正文，再用提供的脚本去验证它。或者生成那条，也就是 ECC 开采你的 repo。你执行 skill-create command 去分析当前的 repo 并产出 skill 文件，而你可以加一个标志，顺便为持续学习生成 instincts。它在本机读取你的 commit 历史，并且抽取可重用的模式。还有一条进阶的 GitHub App 路线，给超大型 repo 和团队分享用。

那么，重点精华。Skills 是标准、耐用的工作流程界面，而新逻辑就属于这里。一个 skill 是一个带主文件的目录，装着一个头部、一个正文，和可选的 codemaps。好的 skills 是严谨的；测试驱动开发工作流程甚至定义了什么才算一个有效的失败测试。有超过 260 个 skills，横跨工程、语言包、上下文工程、学习、编排，和操作者领域。而你用 consult command 去找 skills，并且用手动方式或者 skill-create command 去创建它们。
