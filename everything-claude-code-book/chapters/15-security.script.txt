Chapter fifteen. Security.

Read this chapter before you enable hooks that run shell commands, connect MCP servers, or let an agent loop autonomously. ECC ships a whole security guide for a reason: agentic tooling has become a real attack surface.

Let's be clear about why this isn't paranoia. The security guide opens with a sobering point: the tooling we trust is also the tooling being targeted. Continuous-run harnesses increase the surface area, and prompt injection in an agentic system is no longer a funny jailbreak screenshot. It can become shell execution, secret exposure, workflow abuse, or quiet lateral movement. A few numbers from the guide anchor the stakes, and these are a snapshot. There's a Claude Code pre-trust execution issue rated eight-point-seven on the severity scale. There's a Microsoft writeup on AI memory-poisoning spanning thirty-one companies across fourteen industries. There's a study that found prompt injection in thirty-six percent of nearly four thousand public skills, identifying almost fifteen hundred malicious payloads. And there's a report of more than seventeen thousand exposed agent-harness instances. The exact counts will change. The direction of travel is what matters.

The cleanest mental model is what's called the lethal trifecta, from Simon Willison and cited in the guide. There are three ingredients, dangerous only when combined in the same runtime. First, private data: your secrets, repos, and credentials. Second, untrusted content: a pull-request comment, an email attachment, a fetched web page, an MCP tool's output. And third, external communication: the ability to send data out. Any one or two is usually fine. All three together is when prompt injection becomes data exfiltration. Every byte the model reads is executable context. There's no firm line between data and instructions once text enters the window.

What attack vectors do you actually face? Email and PDF attachments with embedded instructions, which your agent reads as part of a task. GitHub pull requests and issues with hidden diff comments or malicious linked docs, dangerous because automated review bots propagate the exploit downstream. MCP servers that are vulnerable, malicious, or simply over-trusted; OWASP now has an MCP top-ten, covering things like tool poisoning, contextual payload injection, command injection, shadow servers, and secret exposure. Memory poisoning, where a payload gets remembered and resurfaces later. And skills as a supply chain, since a third of public skills carried injection in one study, so treat imported skills like any dependency.

There are real-world vulnerabilities that ended the this-is-overblown phase. Check Point Research disclosed several Claude Code issues. In one, project-contained code could run before the trust dialog was accepted. In another, a malicious project could override the API base-URL variable, redirect API traffic, and leak the API key before trust confirmation. And in a third, repo-controlled MCP settings could auto-approve project MCP servers before the directory was meaningfully trusted. The lesson ECC draws is this: project config, hooks, MCP settings, and environment variables are all part of the execution surface. Your config files are shared through source control and guarded by a trust boundary, which is exactly what attackers target. This is also why you should be cautious about cloning and opening untrusted repos.

The guide is refreshingly concrete about defense in depth. There are five layers, with the easiest return on investment first. First, separate the identity. Don't give the agent your accounts. Create a separate bot user with a short-lived, scoped token. If your agent has the same accounts you do, a compromised agent is you. Second, isolate untrusted work. Run untrusted repos, attachment-heavy tasks, or foreign-content work in a container, a virtual machine, or a devcontainer, and default to no network egress, so a compromised agent can't phone home. Even a plain no-network container for one-off review beats running on your host. Third, restrict tools and paths, which is the highest return and the easiest. If your harness supports permissions, start with deny rules around obvious sensitive material: your SSH keys, your cloud credentials, your environment files, and dangerous shell commands like piping a download straight into a shell, or using remote-access tools. Give a workflow only what it needs. A read-a-repo-and-test job shouldn't read your home directory, and a single-repo token shouldn't have org-wide write. Fourth, sanitize input. Everything the model reads is executable context, so watch for hidden unicode, zero-width or bidirectional characters, payloads hidden in HTML comments, and instructions embedded in attachments or linked content. Sanitize attachments before the model sees them. ECC's prompt-defense baseline is the in-prompt layer of this. And fifth, set approval boundaries, logging, and kill switches. Keep a human in the loop for state-changing actions, log tool and MCP calls, and have a kill switch. Least agency by default.

ECC ships a bundled scanner called AgentShield, a security auditor for your agent configuration that you can run with zero install. There's a quick scan, an auto-fix option for safe issues, a deep three-agent analysis mode, and an init option to generate a secure config from scratch. What does it scan? Your instructions file, your settings, MCP configs, hooks, agent definitions, and skills, across five categories: secrets detection with fourteen patterns, permission auditing, hook-injection analysis, MCP server risk profiling, and agent-config review. The deep mode runs three powerful agents as a red-team, blue-team, and auditor pipeline: the attacker finds exploit chains, the defender evaluates protections, and the auditor synthesizes a prioritized risk assessment. That's adversarial reasoning, not just pattern matching. The output comes as a terminal grade from A to F, plus machine-readable, Markdown, and HTML formats, and it returns a failure exit code on critical findings so you can gate builds. Inside the harness, you run it as a security scan, and in continuous integration, you use the provided action. AgentShield was built at a Claude Code hackathon and reports over twelve hundred tests, around ninety-eight percent coverage, and over a hundred static-analysis rules.

Here's the minimum-bar checklist before you let ECC, or any agent harness, run with real permissions. The agent uses a separate identity, not your personal accounts. Untrusted work runs in an isolated container or virtual machine with no default egress. Deny rules protect your SSH keys, cloud credentials, environment files, and dangerous shell commands. Tools are scoped to what each workflow needs, and tokens are short-lived and least-privilege. Inputs are sanitized, and the prompt-defense baseline is present. State-changing actions need approval, tool and MCP calls are logged, and you have a kill switch. You've run the AgentShield scan and fixed the criticals. You have fewer than ten MCPs and fewer than eighty tools enabled, and you trust each one. And you don't clone and open untrusted repos on your host machine.

ECC isn't just advice about security. It bakes a lot in. The prompt-defense baseline is in every agent and in the main instructions file. Security-first is a core principle, with mandatory pre-commit checks: no hardcoded secrets, input validation, prevention of common web vulnerabilities, authentication checks, rate limiting, and no sensitive data in logs. There's a dedicated security-reviewer agent and a security-review skill aligned with OWASP. There are hooks for secret detection, config protection, governance capture, and MCP health. And there's a supply-chain indicator-of-compromise scan and advisory-source tooling.

So, the key takeaways. The lethal trifecta, private data plus untrusted content plus external communication, is when injection becomes exfiltration. Project config, hooks, MCP settings, and environment variables are execution surface, and real-world vulnerabilities prove it. Defend in depth: separate identity, isolate, restrict tools and paths, sanitize, and approve, log, and kill. Run the AgentShield scan, use the deep mode for adversarial analysis, and gate continuous integration on the failure exit code. And treat skills and MCP servers as supply chain, and don't clone and open untrusted repos on your host.
