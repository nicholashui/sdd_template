第一章。背景和理念。

我们就从 ECC 的由来开始讲。ECC 是 Affaan Mustafa 的作品，他从 Claude Code 在实验阶段推出的时候开始就一直在用。这个项目是从真实的产品工程里成长出来的，最瞩目的就是他做了 zenith dot chat，这个产品赢了 Anthropic 和 Forum Ventures 的黑客松。Repo 自己对它的起源讲得很直白。它说它不只是一堆配置，而是一套完整的系统，在超过十个月密集的每日使用中、一边做真实产品一边演化出来。

这个细节很重要。ECC 不是一份理论上的最佳实践模板，不是谁在一个周末就写得出来的。它是几千个小时代理式编程的化石记录。每一个 hook、每一条 rule、每一个 skill 的存在，都是因为某个真实的失败模式让它变得必要。当你读到一条像“永远不要 commit 一句调试日志”的 rule，或者一个会阻止开发服务器在终端多路复用器以外运行的 hook，你其实是在读一道伤疤。

这个名字是一个变成大项目的小玩笑：Everything Claude Code。它一开始以 Claude 为中心，然后扩展到涵盖整个 AI 编程工具的生态系统。

这里有一条从更新日志整理出来的简短时间线。你不用背它，但它显示了发展的方向。1.2 版统一了 commands 和 skills，加入了会话管理，并且引入了用 instincts 的持续学习。1.3 版带来了完整的 OpenCode 插件支持。1.4 版加入了多语言 rules、一个安装向导，和多代理 commands。1.6 版加入了 Codex 命令行支持、AgentShield 安全审计器，和一个 GitHub Marketplace 应用。1.7 版跨平台扩展，并且加入了一个演示文稿制作 skill。1.8 版把整件事重新定位成一套框架性能系统，有 hook 可靠性大改造和运行时 hook 控制。1.9 版引入了选择性安装、更多语言支持，和一个 SQLite 状态存储。而 2.0 版就变成了代理框架操作系统，有控制平面底座、worktree 生命周期、一个 orchestrator 家族，和一个 Discord 社区。主线很清楚：ECC 从一个配置包，变成了一套用来运行代理的操作系统。

那究竟什么是代理框架，又为什么要优化它呢？框架是一个包住语言模型的程序，让它真正可以做事：读取文件、执行 shell 命令、编辑代码、调用工具。Claude Code、Cursor、Codex 和 OpenCode 全部都是框架。模型是引擎。框架就是包住它的那辆车。

这里是 ECC 建立在上面的关键洞见。代理式编程的质量、成本和安全，大部分是由框架决定的，而不是由模型决定的。两个人用完全相同的模型，得出来的结果可以天差地别，取决于几样东西。模型看到什么上下文，以及它的窗口有多少被浪费了。哪些子任务被委派给更便宜、或者更专门的配置。有什么东西被自动检查，例如测试、格式化和机密信息，相对于有什么东西就只能靠运气。模型在会话之间记得什么。以及模型被允许做什么，也就是工具权限和沙箱。

ECC 是一套框架性能系统：一套经过精心挑选、又经得起实战考验的配置，可以一次性调校以上所有的杠杆。这里有一个好用的比喻。模型是一位出色的新员工。框架是他的笔记本电脑和账号。ECC 就是入职手册、一支导师团队、测试流水线，和安全策略。它是把原始天赋变成可靠产出的一切。

现在我们讲一下五大核心原则。Repo 根目录有三个小小的文件，编码了 ECC 的世界观：一个讲身份，一个讲硬性约束，还有助手会读的指令。有五个原则在到处反复出现。

原则一：代理优先。尽早把工作分派给合适的专家。不是一个通才什么都做，ECC 把工作推给有明确范围的子代理。一个 planner 负责策略，一个 code reviewer 负责质量，一个 security reviewer 负责敏感代码，一个 build-error resolver 在工具链坏了的时候出动。这样就让每一个上下文都小，每一个代理都是专家。

原则二：测试驱动。在信任实现改动之前，先写或者更新测试。ECC 规定要走红、绿、重构的循环，以及在单元、集成和端到端测试上，达到八成以上的覆盖率目标。这是由一个测试驱动开发 skill，以及一些推你去达标覆盖率的 hooks 一起强制执行的。

原则三：安全优先。验证输入、保护机密信息，和保持安全的默认值。每一个代理都带着一条提示防御基线。Hooks 会扫描机密信息。有一份完整的安全指南和一个专门的扫描器，我们会用整个章节去讲它。

原则四：不可变性。宁愿用明确的状态转换，也不要用变更。这是一条编程风格原则：创建新的对象，而不是变更现有的对象。它出现在 rules 里、reviewer 代理的清单里，以及通篇的示例代码里。

原则五：先规划，后执行。复杂的改动应该拆成一个个刻意安排的阶段。plan command 和 planner 代理的存在，就是要阻止助手在没有蓝图的情况下，贸然冲进一个改十二个文件的改动。先规划，拿到确认，然后才动代码。

这五条就是宪法。如果你什么时候想不通为什么 ECC 用某种方式做事，答案几乎一定追溯回其中一条。

还有一套复利工作流程的理念。长篇指南引用了一句话，捕捉到 ECC 最深层的赌注。早期，作者花了时间建立可重用的工作流程和模式。建立它的时候很沉闷，但随着模型和代理框架改进，它产生了一种夸张的复利效应。论点是这样：你今天打的提示用完即弃。但你今天写的 skill 会持续带来回报，每一次会话、每一个项目，而且随着底层模型改进，它会变得越来越好。所以 ECC 大力投资在耐用的产物上：可重用的专家、可重用的工作流程、规划模式，和上下文工程模式，像记忆、压缩，和系统提示注入。这就是为什么技能优先是一句不断重复的口号。一段用完即弃的对话是一支蜡烛。一个 skill 是一个电源插座。

那你实际上会得到什么，用数字来讲呢？在 2.0 版，这个表面是这样，记住这些数目每个星期都会变，所以把它当成一张快照。64 个 agents，是专门做规划、审查、修复构建，和特定语言工作的子代理。262 个 skills，涵盖工作流程和领域知识。84 个 commands，是斜杠项目，再加上一个可选择加入的、已退役垫片的归档区。Hooks，是横跨会话生命周期的自动化。Rules，涵盖通用标准，再加上 TypeScript、Python、Go、Swift、PHP、ArkTS 等等的各语言包。MCP 配置，是连接外部服务的连接器定义。跨平台脚本，用来安装、hooks、会话和编排。一个 Rust 控制平面原型，是 ECC 2.0 的早期版本。还有一个桌面图形界面，让你浏览以上一切。

不要被这些数目吓到。九成时间你只会用到屈指可数的几个 agents，和十几个 skills。其余的是在有冷门需求的时候才出现，无论是 Kotlin 构建错误、HarmonyOS 应用、预测市场研究，还是连视频剪辑都行。是的，真的行。

那么，重点精华。ECC 也就是 Everything Claude Code，一套由超过十个月真实每日使用孕育出来的框架性能系统。框架包住模型，让它可以行动，而 ECC 调校的是框架，不是模型。五个原则贯穿一切：代理优先、测试驱动、安全优先、不可变性，和先规划后执行。ECC 押注在耐用、有复利效应的工作流程上，而不是用完即弃的提示。而那些大数目是一顿自助餐。你只夹你需要的东西。
