Chapter eleven. Everyday workflows.

This is the chapter you'll come back to. It turns all the components into concrete, repeatable loops. Copy these, adapt them, and they become muscle memory.

Let's start with the master loop: plan, then test-driven development, then review, then verify, then learn. Almost every ECC session follows the same shape. Step one is planning. You run the plan command with something like, add OAuth login. The planner agent produces a plan, and you confirm it. Step two is test-driven development. You invoke the test-driven-development workflow skill, going red, green, refactor, guided by the test-driven guide. Step three is review. You run the code-review and security-scan commands, which call the code reviewer and security reviewer. Step four is verify. You run the quality gate, fix any build errors, and confirm test coverage of at least eighty percent. And step five is learn. You run the learn-and-evaluate command and save the session, producing an instinct and a session summary, with the stop hook helping. Around all of it, hooks format, type-check, and secret-scan invisibly, while rules constrain every step. Between phases, clear the context and store intermediate outputs as files, so you don't drown the context window.

Here's the recipe for starting a new feature. You run the plan command with a description like, add user authentication with OAuth. The planner produces a phased implementation blueprint, and you approve it. Then you invoke the test-driven-development workflow: the test-driven guide writes failing tests first, then the minimal implementation. Then you run code-review, and the code reviewer checks the diff, returning zero findings if it's clean. Why this order? Plan-before-execute stops scope sprawl. Tests-first means the implementation is guided by a spec. And reviewing immediately catches issues while the context is fresh.

Here's the recipe for fixing a bug. You invoke the test-driven-development workflow, and the test-driven guide writes a failing test that reproduces the bug, a valid red state. Then you implement the fix and confirm the test goes green. Then you run code-review to catch regressions. The discipline here is the red gate from earlier: you must have a test that genuinely fails for the intended reason before touching production code. No reproduction, no fix.

Here's the recipe for preparing for production. You run a security scan, where the security reviewer does an audit aligned with the OWASP standards, or uses AgentShield. You invoke the end-to-end testing skill, where the end-to-end runner exercises critical user flows in a real browser. You run the test-coverage command to verify eighty-percent-plus coverage. And you run the quality gate as a final check against project standards.

Here's the recipe for orchestrating a bigger build with sub-agents. For multi-phase work, lean on sequential-phase orchestration. Phase one is research, producing a research summary, using a cheap model. Phase two is planning, producing a plan, using a powerful model for hard design. Phase three is implementation, producing code and tests, on a balanced model. Phase four is review, producing a review. And phase five is verify, which either finishes or loops back to the failing phase. The orchestration commands package these patterns, for building an MVP, adding a feature, fixing a defect, and so on. And for multi-model collaboration, the multi commands exist, though remember they need their separate workflow runtime installed.

Let's talk about parallelization: doing more at once. When one session isn't enough, ECC's guidance is deliberate, not just spin up ten terminals. First, fork for non-overlapping work. Use a fork command to branch a conversation for questions or research, while your main chat keeps editing code. Keep code changes in the main chat, and use forks for read-only exploration. Second, use git worktrees for overlapping code work. When multiple instances must edit overlapping code, give each its own checkout, and run one instance per worktree, naming your chats so you don't lose track. Third, there's the cascade method: open new tasks in tabs to the right, sweep left to right from oldest to newest, and focus on at most three or four tasks at a time. The guiding principle is this question: how much can you get done with the minimum viable amount of parallelization? Add a terminal only out of true necessity.

Here's a nice pattern for greenfield work: the two-instance project kickoff, using two assistant instances on an empty repo. The first instance is a scaffolding agent that lays down the project structure and configs. The second instance is a deep-research agent that connects to services and web search, writes the product-requirements document, draws architecture diagrams, and collects real documentation. Then they converge: the scaffold meets the researched plan.

Then there's the end-of-day handoff. Don't let a session evaporate. Capture it. Run the learn-and-evaluate command to extract reusable patterns and self-evaluate them before saving, then save the session to snapshot the state. Or rely on the stop hook to write a summary automatically. The next morning, resume the session to reload the latest one and continue. And for a portable, shareable status, great for standups or handing off to a teammate, there's a status command that writes a markdown handoff file.

ECC treats verification as a first-class workflow, not an afterthought. There are two eval styles. Checkpoint-based, where you set explicit checkpoints, verify against criteria, and fix before proceeding. And continuous, where you run every few minutes or after major changes, with the full test suite and linter. There are also two ways to read reliability. The first is pass-at-k, where at least one of k attempts succeeds, which you use when you just need it to work. The second is pass-to-the-k, where all k attempts succeed, which you use when consistency is essential. The evaluation-harness and verification-loop skills implement these, and the quality gate is the everyday gate.

A few high-leverage habits from the guides, small things with big payoff. Disable auto-compaction, and compact strategically at logical breaks. Run long commands in a terminal multiplexer so you can detach, reattach, and stream logs. ECC even reminds you. Use semantic search over plain text search to halve search tokens. Set up a status line showing your branch, context percentage, model, and to-dos. Keep MCPs lean per project. And review diffs before committing, letting the git-push hook gate you.

So, the key takeaways. The master loop is plan, test-driven development, review, verify, learn, all wrapped in hooks and rules. Reproduce bugs with a valid failing test before fixing. Orchestrate big work in sequential phases, passing files between agents. Parallelize with forks, git worktrees, and the cascade method, but minimally. Capture every session so tomorrow starts ahead. And verify with checkpoint and continuous evals, and pass-at-k versus pass-to-the-k thinking.
