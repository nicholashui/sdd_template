第十四章。Token 优化和性能。

代理式编程可以很快变得昂贵，而且当你填满上下文窗口，质量其实会下降。ECC 把成本和上下文当成有真实杠杆的工程问题。这一章把它们收集在一起。

我们从真正有作用的设置开始讲。在你的 Claude 配置文件里面，有几个选择最重要。模型设置默认是最强大的模型，但建议是用平衡的模型。这样可以削减大约六成成本，并且应付超过八成的编程任务。最大思考 token 设置默认接近三万二千，但把它降到一万，就削减了每个请求大约七成的隐藏思考成本。自动压缩百分比默认是九十五，但把它设成五十，就会更早压缩，在长会话里面给出更好的质量。而订阅用户可以关闭成本警告，这样就隐藏了 API 费率的估算，同时保留上下文、范围和循环的警告。只在你需要深度的时候，才升级到最强大的模型。

唯一最大的成本杠杆，是模型路由：让模型对应到任务。换句话说，就是不要什么都用最强大的模型。这里是 ECC 推荐的路由。至于探索和搜索，用最便宜的模型，因为它快、便宜，而且找文件绰绰有余。至于简单的单文件编辑，又是最便宜的模型，因为指令清晰而风险低。至于多文件实现，平衡的模型，是编程的最佳平衡。至于拉取请求审查，平衡的模型，可以负担得起地捉到细微之处。至于复杂的架构，最强大的模型，做深度推理。至于安全分析，最强大的模型，因为你输不起漏东西。至于调试困难的 bug，最强大的模型，因为你必须把整个系统记在脑里。而至于写文档，最便宜的模型，因为结构简单。重点：大约九成的编程，默认用平衡的模型。只在第一次尝试失败、任务横跨五个或者以上文件、是一个架构决定，或者是安全关键的时候，才升级到最强大的。这也是为什么 sub-agents 这么重要：一个 sub-agent 架构，让你可以为每一个子任务指派最便宜但足够的模型，而不用为一次文件搜索付顶级费率。ECC 的 model-route command 和 harness-optimizer agent，帮你把这样东西自动化。

接着，保护上下文窗口，因为在接近上限那里，质量会跌下悬崖。用几个方法守住它。为大型重构和多文件功能，避开窗口最后的大约两成，不过单次编辑和文档可以容忍更高的使用率。保持 MCP 精简，启用少于十个服务器和少于八十个工具，单是这样就可以收回超过十万个 token。封顶会话开始的上下文，或者为本机、低上下文模型完全关闭它。用策略性压缩：停用自动压缩，并且在合乎逻辑的断点压缩。ECC 有一个推你一把的 hook，而压缩前 hook 会先保存状态。用 context-budget command 去看 token 去了哪并且削减开销。并且依靠上下文监控，也就是那个在耗尽、失控成本、范围蔓延和工具循环警告你的 hook。


还有工具层级的 token 节省。用语义搜索代替纯文本搜索。在 ECC 的五十项任务基准测试里面，配合 Claude Code 的语义搜索，在相似或者更好的评定质量下，用了大约一半的 token。保持代码库模块化，文件是数百行而不是数千行，这样读取更便宜，也更常一击即中，而这也是一条 rule。用一个包住命令行工具的 skill，代替一个总是存在的 MCP，这样可以把工具描述挡在上下文之外。并且用 codemaps，让 skills 携带轻量的地图，这样助手就不用重新探索地去导航。

至于并行化，也就是日常工作流程那一章讲过的，总结是这样：并行工作让吞吐量倍增，但 ECC 建议最低可行的量。为不重叠、只读的工作用分叉，像问题和研究，而主对话编辑代码。为重叠的代码工作用 git worktrees，每个检出一个实例，并且为你的对话重新命名。并且用级联方法，最多聚焦在三四个任务，从最旧到最新。反对只因为你做得到就开十个终端的指引是：每一个额外的实例，都是你要管理的上下文。只在必要时才加一个。

还有验证效率。因为你没验证而要重做工作，本身就是一种成本。ECC 的评估定位：在合乎逻辑的里程碑运行检查点评估，并且在重大改动之后运行持续评估。在你只需要它跑得起来的时候用 pass-at-k，在你需要一致性的时候用 pass-to-the-k。评估框架和验证循环 skills，加质量关卡，实现了这样东西。早点捉到一个回归，比它出货之后才捉到便宜很多。

至于成本可见度，有一个 cost-report command 给一份成本摘要，和 context-budget command 做 token 开销分析。仪表板和 status 工具拿出度量，这样成本就不再是一个谜，而一个 metrics-bridge hook 为状态栏和上下文监控保持一个运行中的会话聚合。

如果你想要一个单一、又快又便宜的起步配置来开始，这里用文字讲出来。在你的设置，用平衡的模型、把最大思考 token 设成一万，并且把自动压缩设成五成。然后把你的 hook 配置设成 standard、封顶会话开始的上下文、逐个项目保持启用少于十个 MCP 和少于八十个工具、默认用平衡的模型而只在真正需要时才升级到最强大的、用语义搜索、保持文件模块化，并且在合乎逻辑的断点压缩。

那么，重点精华。设置平衡的模型、一万个思考 token，和五成自动压缩，去大幅削减成本。按任务路由模型：搜索用最便宜的、平衡的做默认、最强大的只用在困难或者关键的工作。保护上下文：少于十个 MCP 和八十个工具、封顶会话开始、策略性地压缩，并且留意 context-budget。语义搜索和模块化文件，实质地减少 token 使用。并且最低限度地并行化，同时早点验证，用检查点和持续评估和 pass-at-k 对 pass-to-the-k 的思维。
