第十章。MCP 和上下文管理。

我们从 MCP 是什么开始讲。MCP，也就是 Model Context Protocol，是一个标准的方式，把你的助手连接到外部服务：GitHub、一个数据库、一个部署平台、一个浏览器。一个 MCP 服务器暴露出模型可以直接调用的工具。不用你把一个数据库结果复制粘贴到对话，模型自己会查询那个数据库。短篇指南讲得很好：一个 MCP 是一个由提示驱动、包住一个 API 的包装。它不是 API 的替代品，而是模型导航它的一个灵活方式。例如，一个数据库 MCP 让助手可以列出数据表并且执行查询，不用离开会话。

ECC 把它的 MCP 配置放在一个文件，并且恰好内置一个默认连接器，给 Chrome 开发者工具用，提供浏览器控制。其余一切都是可选择加入，要么是一个包住命令行工具或者 REST API 的 skill，要么就是一个你刻意启用的目录项目。这是一个有记录的策略，它的存在是为了保护这一章真正要讲的东西：你的上下文窗口。

这里是整个 ECC 里面最重要的操作教训，它让差不多每个人都意外，所以读两次。每一个 MCP 工具描述，在你开始做任何事之前，就已经在消耗你上下文窗口的 token。太多启用了的 MCP，可以把一个二十万 token 的窗口缩到大约七万，而当你接近上限，质量会剧烈下降。每一个启用了的 MCP 服务器都会宣传它的工具，而每一个工具的描述都永久地坐在上下文里面。启用十几个多嘴的 MCP 服务器，你就花了三分之一的窗口在那些你可能永远都不会用的菜单上面。

所以这里是经验法则，直接从指南来。保持启用少于十个 MCP 服务器。保持少于八十个工具活跃。配置了二三十个 MCP 完全没问题。只逐个项目停用你没用的那些就行。你可以用 MCP command 看一下启用了什么，并且在那里停用没用的那些。一个提示：MCP command 是可靠的实时开关。编辑你的配置文件，不是一个可靠的方法去关闭已经加载的 MCP 服务器，所以用 MCP command。


长篇指南分享了一个犀利的优化：用命令行工具代替 MCP 的技巧。很多 MCP 是可以替代的。像 GitHub 的版本控制、数据库，和部署平台，已经有出色的命令行工具，而 MCP 本质上就是在包住它们。所以不要总是加载一个 MCP，而是创建一个包住你需要那个命令行工具的 skill 或者 command。例如，一个用你偏好的标志去执行 GitHub CLI 的拉取请求 command。你得到方便，又没了那个总是存在的上下文成本。这是任何有好命令行工具的东西的推荐默认。把 MCP 留给那些工具调用人体工学真的有帮助的服务。

你在 ECC 里面怎样配置 MCP 呢？因为插件安装不自动启用内置的 MCP，所以你要选择加入。你可以在 Claude Code 里面用 MCP command 实时做，它会加入、启用或者停用服务器并且留存那个改动。或者，为了 repo 本地访问，你把你想要的服务器复制进一个项目配置文件，并且把任何占位密钥换成真正的。而如果你已经在跑自己那份 ECC 内置的 MCP，你就设置一个列出它们的环境变量，这样安装和同步流程就会跳过它们，而不是再加重复的。那个环境变量是一个安装和同步的过滤器，不是一个实时开关。ECC 知道的那套 MCP 服务器，也就是你逐个项目只会启用屈指可数那些，包括 GitHub、一个数据库连接器、一个实时文档服务、一个神经搜索服务、记忆、一个浏览器自动化工具、一个按顺序思考的助手、几个部署平台、一个分析数据库、一个网络爬取服务，和默认的 Chrome 开发者工具连接器。

不过上下文纪律不只是关于 MCP。ECC 和那些指南推荐了几个其他习惯。第一，用一个语义搜索工具，代替纯文本搜索。在 ECC 的五十项任务基准测试里面，语义搜索在相似或者更好的质量下，用了大约一半的 token。第二，保持代码库模块化，文件是数百行而不是数千行。小一点的文件意味着更便宜的读取和更好的一击即中，而这也是一条 rule。第三，用策略性压缩：停用自动压缩，改为在合乎逻辑的间隔压缩。ECC 有一个在好时机推你一把的 hook，而压缩前 hook 会先保存状态。第四，依靠上下文监控，也就是那个在上下文耗尽、失控成本、范围蔓延和工具循环咬到你之前警告你的 hook。而第五，用 context-budget command 去分析你的 token 去了哪，并且削减开销。会话开始的上下文注入本身就有封顶，在一个合理的默认值，而你可以调校它，或者为低上下文的配置完全关闭它。

有一个 MCP 的近亲值得一提：插件。插件打包工具，方便安装，而一个单一的插件可以捆绑 skills、hooks、MCP 服务器，或者语言服务器集成。同样的上下文警告适用。启用很多插件，连同它们的工具，会吃你的窗口。作者保持大约十四个插件装着，但一次只启用四五个。特别有用的插件类型，包括语言服务器插件，在你在 IDE 以外运行助手的时候，给你实时类型检查和跳到定义；hookify 插件，用来用对话方式创建 hooks；语义搜索插件；和一个实时文档插件。

那么，重点精华。MCP 把模型连接到外部服务，做可调用的工具，而 ECC 默认只内置 Chrome 开发者工具连接器。每一个启用了的 MCP 都会预先花费上下文 token，所以保持启用少于十个服务器和少于八十个工具：配置多些，启用少些。用 MCP command 做实时开关，不是用配置文件。对于有好命令行工具的服务，偏向用一个包住命令行工具的 skill，多过一个总是存在的 MCP。并且用语义搜索、模块化文件、策略性压缩、上下文监控，和 context-budget command 来保护你的上下文。

现在我们把以上一切，拼成你实际上每天会运行的工作流程。
