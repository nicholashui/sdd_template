Chapter three. Installation and setup.

This is the chapter to get exactly right, because a sloppy install is the number one cause of ECC feeling broken or duplicated. The good news is that the rules are simple.

Let's start with the one rule that prevents most problems. Pick exactly one install path, and never stack them. There are two paths. The first is the plugin, which is the recommended default for Claude Code. The second is the manual installer, or command-line interface. The classic broken setup is to run the plugin install, and then also run the full installer. The plugin already loads ECC's skills, commands, and hooks. The full installer then copies the same things into your user directories, giving you duplicate skills and duplicate runtime behavior. If you've already done this, your first stop should be the reset section later in this chapter, before you do anything else.

Now, the requirements. You'll need the Claude Code command-line tool at version two-point-one-point-oh or later, because the plugin's hook auto-loading depends on it. You'll need Node, since the hooks and scripts are Node-based and run cross-platform on Windows, macOS, and Linux. And you'll need a package manager, whether that's npm, pnpm, yarn, or bun. ECC auto-detects which one you use.

Let's walk through path A, the plugin, which is recommended for Claude Code. Inside a Claude Code session, you add the marketplace pointing at the ECC repository, then install the ecc plugin. You can also wire it up declaratively in your Claude settings file, by listing the marketplace and enabling the plugin there. This gives you the agents, skills, commands, and hooks instantly. But not the rules, and we'll come back to that.

There's an important hook gotcha here, mainly for contributors. Do not add a hooks field to the plugin's manifest file. Claude Code version two-point-one and later auto-loads a plugin's hooks configuration by convention, and declaring it again causes a duplicate-hooks-file error. There's even a regression test guarding against this, precisely because it bit the project repeatedly. And if hooks feel too intrusive, or you only want rules, agents, commands, and core skills, you can skip the plugin and use the minimal manual profile instead. Hooks are intentionally excluded from the minimal profile.

Now path B, the manual installer. You clone the repository, install dependencies, then run the installer with a profile and a target. For example, you might install the minimal profile targeting Claude Code. On Windows, there's a PowerShell version, and you can even run it without cloning by using the npx installer.

Let's talk about profiles. The minimal profile gives you rules, agents, commands, and core workflow skills, but no hooks runtime. The core profile gives you the standard set, and you can keep core while disabling hooks with a flag. The full profile gives you everything, and you should use this only on the fully manual path, never after a plugin install.

ECC also has a manifest-driven selective installer, so you can add or remove components precisely. You can add just the hooks runtime later. You can keep core but skip hooks. Or you can add a specific capability, like machine learning, on top of minimal. And if you don't know what to install, ask the advisor. There's a consult command where you describe a need, like security reviews, or model training and deployment, and it returns matching components, related profiles, and preview and install commands. You can preview before installing if you want to inspect the exact file plan.

Here's something that trips everyone up: rules need a manual copy, always. Claude Code plugins cannot distribute rules. That's an upstream limitation. So regardless of which path you took, if you want ECC's always-follow guidelines, you copy them yourself. You can install them user-level, applying to all projects, or project-level, applying only to one project. A few rules of thumb. Start with the common rules plus one language pack you actually use. Copy whole language directories, not the individual files inside them, so that relative references and filenames don't collide. And don't copy every rules directory unless you genuinely want all that context loaded.

If you're on the manual path, install hooks correctly. Do not hand-copy the repository's hooks configuration into your settings file. That file is plugin- and repo-oriented, and raw copying isn't supported. Use the installer instead, so the command paths get rewritten correctly. The installer writes resolved hooks into the right location and leaves your settings file untouched. And if you used the plugin, you already have hooks auto-loaded, so do not also copy them into your settings.

Now, resetting and uninstalling. If ECC feels duplicated, intrusive, or broken, do not reinstall on top of itself. Diagnose and clean first. Start with the lifecycle wrapper. You can list what's installed, run a doctor command to diagnose, run a repair command to restore ECC-managed files, and run uninstall in dry-run mode to preview. There's also a direct uninstall script with a preview option. ECC only removes files recorded in its install state, so it won't delete unrelated files. If you stacked methods, clean up in this order. First, remove the Claude Code plugin install. Second, run the ECC uninstall from the repo root, which removes state-managed files. Third, delete any extra rule folders you copied manually. And fourth, reinstall once, using a single path. One reassurance: if your local Claude setup was wiped, you do not need to repurchase anything. Just run list-installed, then doctor, then repair, before reinstalling.

A word on MCP servers: they're opt-in. Plugin installs intentionally do not auto-enable ECC's bundled MCP server definitions. This avoids overlong tool names on strict gateways and protects your context window. ECC ships exactly one default connector, for Chrome developer tools. Everything else is either a skill wrapping a command-line tool or a REST API, or an opt-in catalog entry. To add MCP servers, you can use Claude Code's MCP command for live changes, or copy the servers you want into a project config file for repo-local access. If you already run your own copies, you can set an environment variable so the install and sync flows skip them. And always replace any placeholder keys with real ones. Context-window discipline gets its own chapter later.

One more setup note: multi-model commands need extra setup. The multi-plan, multi-execute, multi-backend, multi-frontend, and multi-workflow commands are not covered by the base install. They depend on a separate workflow runtime that you install with its own npx command. That provides the external dependencies these commands expect. Without it, the multi commands won't run correctly.

To verify your install, you can list what the plugin provides from inside Claude Code, run the test suite from the repository, or take an operator readiness snapshot that writes a portable handoff file. A successful manual install will show ECC-managed files when you run the list-installed command.

Finally, a few quick pointers for other harnesses. Full cross-harness detail comes later, but the target flag is the key. You can target Cursor, target Zed with the minimal profile, run a sync script to merge into Codex, or simply launch OpenCode, which auto-detects its own config directory.

So, the key takeaways. One path only: plugin or manual installer, never both. The plugin is fastest, but rules still need a manual copy, the common rules plus one language pack. The profiles are minimal with no hooks, core, and full, where full is manual-only. Use the consult command if you're unsure what to install. Reset with list-installed, doctor, and repair, then a dry-run uninstall, before reinstalling. And MCP servers and the multi commands are deliberately opt-in.
