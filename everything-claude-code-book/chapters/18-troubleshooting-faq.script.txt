Chapter eighteen. Troubleshooting and frequently asked questions.

This is a field guide to the problems you'll actually hit, ordered roughly by how common they are. The repo's own troubleshooting doc and frequently-asked-questions informed this chapter.

First problem: everything looks duplicated, or ECC feels intrusive or broken. The cause is that you stacked install methods, running the plugin and the full installer. The plugin already loads skills, commands, and hooks, and the installer copied them again. Here's the fix, in order. List what's installed. Run the doctor command to diagnose. Run the repair command to restore managed files. And if it's still messy, run the uninstall, first in dry-run mode, then for real. After that, remove the plugin install, uninstall via the repo, delete extra manually-copied rule folders, and reinstall once, one path only.

Second problem: a duplicate-hooks-file error, or hooks aren't working. The cause is that Claude Code version two-point-one and later auto-loads a plugin's hooks configuration. You either declared a hooks field in the plugin manifest, or copied the hooks configuration into your settings file on top of a plugin install. The fix: don't add a hooks field to the plugin manifest, a regression test enforces this; plugin users shouldn't copy hooks into their settings file; and manual users should install hooks through the installer with the hooks-runtime module.

Third problem: your context window is shrinking, or the assistant runs out of context. The cause is too many MCP servers and tools enabled, since each tool description eats your window. A two-hundred-thousand-token window can drop to around seventy thousand. The fix: use the MCP command to disable unused servers. Keep fewer than ten MCPs and fewer than eighty tools. Also cap the session-start context, or turn it off, compact strategically, and check the context budget. And remember, editing your settings file is not a reliable way to disable already-loaded MCPs, so use the MCP command.

Fourth problem: hooks are too noisy, too strict, or too slow. Tune them at runtime, with no file editing. Set the hook profile to minimal for quieter, or strict for maximum. Disable specific hooks by their ids. Or keep the warnings but drop the cost estimates. Alternatively, install the minimal profile, which has no hooks runtime, and add hooks back later with the hooks-runtime module.

Fifth problem: you installed the plugin but you have no rules. This is expected. Claude Code plugins cannot distribute rules. Copy them manually: clone the repository, make a rules directory under your Claude config, and copy in the common rules plus your stack's language pack. Copy whole language directories, not individual files.

Sixth problem: the multi commands don't work. The cause is that they need an external workflow runtime that the base install doesn't provide. The fix is to run that runtime's setup through its npx command, which sets up the wrapper and prompt files those commands expect.

Seventh problem: skills or commands aren't showing up. Check the install, by listing the plugin's contents or listing what was installed manually. Check skill placement, because Claude Code loads skills only from direct children of your skills directory, so don't nest them in a subfolder. Check your version, ensuring the command-line tool is at least version two-point-one-point-oh. And note that the Codex plugin path is fragile upstream, so prefer the sync script over the Codex plugin marketplace.

Eighth problem: your local Claude setup got wiped, and you wonder if you need to repurchase ECC. No. Run the list-installed command, then doctor and repair, before reinstalling, and that usually restores ECC-managed files. Billing and account recovery for the hosted tools is a separate concern from the open-source files.

Ninth problem: Cursor and Claude Code are overwriting each other's memory. Set a separate agent data home for Cursor, using an environment variable. Cursor installs try to do this automatically through a session-start hook and an override file. And if you deliberately want to share memory, point both at your Claude directory.

Now a quick frequently-asked-questions round. Can you use only some components, like just agents? Yes. Do a manual install and copy what you want, since each component is independent. Does it work with your IDE or command-line tool? It supports Claude Code natively, plus Cursor, Codex in both app and command-line form, OpenCode, Gemini, Zed, GitHub Copilot, Antigravity, JoyCode and CodeBuddy, Qwen, and a manual path for others. Can you run ECC on a custom API endpoint? Yes, set the base-URL and auth-token environment variables, since ECC is provider-agnostic once the assistant works, but be aware of the base-URL hijack vulnerability, so only trust your own gateway. How do you check what's installed? List the plugin's contents, or run the list-installed command. How do you contribute a skill or agent? Fork the repo, add your skill or agent file with the proper header, run the tests, and open a pull request. And is it really free? The repo is MIT-licensed forever; the hosted tools and sponsorships fund development but aren't required.

There's a handy first-thing-to-run sequence when something's weird: list what's installed, run doctor to see what's wrong, run repair to fix managed files, run the test suite to check whether the repo itself is healthy, and take a full readiness snapshot with the status command.

So, the key takeaways. Most problems trace to stacked installs or double-loaded hooks, so diagnose with doctor and repair, and never reinstall on top. Context issues mean trimming MCPs and capping the session-start context. Missing rules is expected with the plugin, so copy them manually. The multi commands need their separate runtime. And when in doubt: list-installed, then doctor, then repair.
