第十二章。跨框架使用。

这里是大概念：写一次，到处运行。虽然名字叫 Everything Claude Code，但 ECC 是多工具的。它自称是第一个把每一款主要 AI 编程工具发挥到极致的插件。你编写你的 agents、skills 和 rules 一次，而安装器就把它们翻译成每一款框架的原生布局。

有两个架构决定让这样东西得以实现。第一，在 repo 根目录有一个通用上下文文件，给 Claude Code、Cursor、Codex 和 OpenCode 一起读取，而 GitHub Copilot 就改用它自己的指令文件。第二，一个“不要重复你自己”的适配器模式，让其他框架可以重用 Claude Code 的 hook 脚本，而不用复制它们。例如，Cursor 的适配器把 Cursor 的事件数据，转换成 Claude 的 hook 脚本预期的形状。安装目标只是一个标志，在那里你挑 Claude、Cursor、Codex、OpenCode、Zed 等等。

我们看一下框架计分卡，因为功能覆盖因工具而异，而这些数目是一张会变的快照。Claude Code 是最丰富的，有 64 个 agents、84 个 commands、262 个 skills、八个 hook 事件、34 条 rules、十四个 MCP 连接器，和它自己的上下文文件。Cursor 共用 agents、commands 和 skills，有更多的 hook 事件，达到十五个，用它自己的前言区格式携带 rules，并且用通用上下文文件。Codex，无论是它的应用还是命令行形式，都共用 agents，并且对 commands 和 rules 用一个基于指令的做法，暂时没有 hook 事件，并且用一个配置文件来处理它那屈指可数的 MCP 服务器。OpenCode 有它自己的原生一套：十二个 agents、35 个 commands、37 个 skills、十一个 hook 事件、十三条 rules，和完整的 MCP 支持。而 GitHub Copilot 是最轻量的，有屈指可数的提示、基于指令的 skills 和 rules、没有 hook 事件，和一个总是存在的单一指令文件。可以这样读：Claude Code 是参考目标，有最丰富的界面。Cursor 和 OpenCode 其实有更多的 hook 事件。而 Codex 和 Copilot 没有一个 hook 系统，所以那里的强制执行是基于指令的，而不是运行时。

Claude Code 是主要的原生目标。本书里面所有东西都直接适用。你通过插件或者手动安装器去安装，而它需要命令行工具是 2.1.0 版或者之后。


至于 Cursor，预先翻译好的配置住在它的配置目录，而你通过把目标设成 Cursor 和你的语言去安装。你会得到：十五个 hook 事件、十六个薄薄的 hook 脚本，通过一个适配器委派给共用脚本、用 Cursor 前言区格式的 34 条 rules、用一个前缀安装、避免碰撞的 agents，再加共用的 skills、commands 和 MCP。记忆隔离在这里很重要。因为 Cursor 重用同样的 hook 脚本，ECC 自动把 Cursor 的记忆隔离在你的 Claude 目录之外，用一个设置独立数据主目录的会话开始 hook，连同一个默认位置、一个覆写文件，和一条总是生效的 rule。如果你刻意想和 Claude Code 共用记忆，你就把数据主目录变量指向你的 Claude 目录。Cursor 的 hook 系统其实比 Claude 的更丰富，有像 shell 执行前、文件编辑后、用来检测机密信息的提交提示前、标签文件读取前的事件，后者会阻止标签功能读取像环境和密钥文件这样的敏感文件，还有 MCP 执行前后的事件用来做审计记录。

至于 Codex，也就是 macOS 应用和命令行工具，通过通用上下文文件加一个配置文件，有一等的支持。最干净的设置是同步脚本，它会对你的 Codex 配置做一个只加入的合并，有一个预览选项，和一个刷新服务器的标志。或者你可以只在 repo 里面跑 Codex，因为那个上下文文件和配置目录会自动检测。参考配置刻意不钉死模型或者供应商，因为 Codex 用它自己的默认。包括了什么：顶层的批准、沙箱和网络搜索配置、六个 MCP 服务器，或者加了一个数据库就七个、32 个 skills、两个配置文件，一个严格只读的和一个全自动批准的，和三个示例 agent 角色，一个探索员、一个 reviewer，和一个文档研究员。关键限制：Codex 暂时没有 hook 执行的对等，所以 ECC 的强制执行是基于指令的，通过那个上下文文件、可选的指令覆写，和沙箱和批准设置。实验性的 Codex 插件 marketplace 路线，在上游很脆弱，所以偏向用同步脚本。

至于 OpenCode，有完整的插件支持。你只在 repo 里面跑它，它就会自动检测它的配置。或者你从 npm 安装已发布的插件，并且在 OpenCode 配置里面引用它。注意 npm 插件启用了 ECC 的 OpenCode hooks、事件和插件工具，但它不会自动加入完整的 command、agent 和指令目录。要完整的设置，在 repo 里面跑 OpenCode，或者把配置资产复制进你的项目，并且接好那些指令、agents 和 commands。OpenCode 的插件系统很精密，有超过二十种事件类型。ECC 把 Claude 的 hooks 对应到 OpenCode 事件，并且加入额外的，例如文件已编辑和语言服务器诊断。它还暴露了六个原生自定义工具，例如 run-tests、check-coverage，和 security-audit。

至于在 VS Code 的 GitHub Copilot，不用额外工具。Copilot Chat 自动读取指令和提示文件。有一个总是注入的指令文件，有编程风格、安全、测试和 git 的核心 rules。有可重用、按需的提示文件，给像 plan、测试驱动开发、安全审查、build 修复和重构的东西用。还有一个配置文件，有各任务的指令叠加和提示文件的启用。限制：Copilot 没有一个 hook 系统，也没有 subagent API，所以 ECC 的自动化和代理委派不适用。但你在每一个对话里面，仍然得到完整的编程理念。

还有一条长尾的其他工具。有给 Gemini 命令行工具的实验性项目本地支持。有一个保守的 Zed 适配器，在那里你把 API 密钥放在 Zed 自己的设置，而不是 repo。有给 Antigravity 的紧密集成设置。有给 JoyCode 和 CodeBuddy 的项目本地选择性安装适配器。有给 Qwen 命令行工具的主目录适配器。而对于像 Grok 的非原生工具，有一条手动后备路线。这些每一个在文档里面都有它自己的指南。

你也可以在一个自定义端点或者网关上跑 ECC。ECC 不硬编码 Anthropic 的传输，所以它通过 Claude Code 正常的网关支持运作。你设置一个 base-URL 环境变量和一个 auth-token 变量，然后启动。如果你的网关重新对应模型名，你在 Claude Code 里面设置那样东西，而不是在 ECC，因为一旦助手跑得起来，ECC 的 skills、hooks、commands 和 rules 就和供应商无关。一个警告：一个恶意项目覆写那个 base-URL 变量，曾经是一个真实世界的漏洞，我们会在安全那一章讲。

那么，重点精华。ECC 是多框架的：编写一次，而安装器去适配它，只改那个目标标志。通用上下文文件是共用的上下文，而一个“不要重复你自己”的适配器，在各工具之间重用 hook 脚本。Claude Code 是参考，Cursor 和 OpenCode 有更多 hook 事件，而 Codex 和 Copilot 没有 hooks，所以那里的强制执行是基于指令的。用数据主目录变量，去在各框架之间隔离记忆。而那些设置路线是：Codex 用一个同步脚本，OpenCode 用 repo 内运行或者 npm 插件，而 Copilot 用指令和提示文件。
