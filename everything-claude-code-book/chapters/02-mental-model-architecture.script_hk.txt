第二章。心智模型同架構。

要理解 ECC，而唔係淨係死記，最快嘅方法就係將佢睇成四個疊起嚟嘅層。每一層回答一個唔同嘅問題，而每一層都係由佢上面嗰層塑造出嚟。

第一層係身份同管治。呢個就係 ECC 係邊個。三個細細嘅根目錄檔案訂立咗憲法。有一個靈魂檔案，裝住核心身份同五大原則。有一個 rules 檔案，裝住硬性嘅「永遠要」同「永遠唔好」約束，再加上 agents、skills 同 hooks 嘅必要格式。仲有助手會讀嘅指令檔案，包括代理名冊、安全指引同工作流程。呢一層就係點解其他所有嘢會咁樣運作嘅原因。

第二層係組件目錄。呢個係可重用嘅嘢，係 repo 嘅核心，係你會安裝同呼叫嘅嘢。佢分成 agents、skills、commands、rules，同 MCP 設定。本書大部分內容都係講呢一層。

第三層係框架轉接器同安裝器。呢個就係佢點樣落到磁碟上面。目錄係通用嘅，但每一款 AI 工具都想佢嘅設定放喺唔同嘅位置，用唔同嘅格式，所以呢一層負責翻譯。有針對唔同平台嘅安裝入口。有一個由清單同安裝腳本驅動嘅選擇性安裝引擎。一個狀態儲存記錄咗裝咗啲乜，呢個令到乾淨嘅更新同解除安裝變得可能。而目標就係各工具嘅設定目錄，畀 Claude、Cursor、Codex、OpenCode、Gemini、Zed、GitHub 等等用。

第四層係運行時同強制執行。呢個就係你工作嗰陣發生緊嘅嘢。當一個會話真正運行緊，呢一層就係生勾勾。一個 hooks 設定將事件對應到細細嘅腳本。嗰啲 hooks 會格式化程式碼、掃描秘密資料、把守品質、留存記憶，同埋捕捉學習訊號。而度量同可觀測性，就餵畀操作者命令列介面同 ECC 2.0 嘅控制平面。

呢度有一個記住以上一切嘅心智捷徑。身份管治目錄。目錄由轉接器塑造。轉接器將佢落到磁碟。而運行時負責強制執行。上層管治下層。

而家我哋遊覽吓 repo 嘅地圖。你唔會直接掂到大部分嘅嘢，但了解吓地形，可以消除好多神秘感。喺根目錄，你有身份同管治嘅檔案，再加一個版本檔案。然後就係目錄裏面嘅各個資料夾：agents，有 64 個專門嘅子代理；skills，有超過 260 個，每個都喺自己嘅資料夾；commands，有 84 個斜線 commands，保持向後相容但係技能優先；一個可選擇加入嘅、已退役短 command 名嘅封存區；rules，有通用 rules 加各語言包；hooks；MCP 設定；同 contexts，會喺開發、審查同研究模式裏面，將內容注入系統提示。

仲有一個 scripts 資料夾，裝住跨平台嘅管道工程：真正嘅 hook 實作、共用工具、操作者命令列介面，同選擇性安裝嘅規劃器同套用器。有一個畀 ECC 2.0 用嘅 Rust 控制平面原型。有一個桌面圖形介面。有安裝器入口、安裝清單、用嚟驗證嘅 JSON 結構、一套測試，同埋好似 SaaS、Go、Django 同 Rust 專案咁嘅範例設定。

Repo 入面有三份指南係純金。Repo 講得好白：呢個 repo 淨係原始程式碼，而啲指南會解釋一切。短篇指南涵蓋基礎、skills、hooks、subagents、MCP 同外掛，你應該先睇佢。長篇指南涵蓋 token 經濟學、記憶留存、評估，同並行化。而安全指南涵蓋攻擊向量、沙箱、真實世界嘅漏洞，同 AgentShield。本書將呢三份提煉同整理，但原作本身就好出色。

我哋追蹤吓一個單一請求點樣流過 ECC。假設你執行 plan command，請求加入 OAuth 登入。呢度就係實際發生嘅嘢。第一，管治載入。Rules 同指令檔案已經喺上下文裏面，所以助手知道啲標準：測試驅動開發、不可變性、安全。第二，如果有裝，一個會話開始 hook 已經注入咗之前會話嘅記憶，並且偵測咗你嘅套件管理器。第三，command 或者 skill 啟動。Plan command 對應到規劃邏輯，而呢個邏輯可能會委派畀 planner 代理。第四，代理喺一個有明確範圍嘅上下文裏面工作，用有限嘅工具同啱嘅模型，然後交返一個計劃。第五，你確認，因為先規劃後執行就意味住佢會等你。第六，實作開始進行，好可能透過測試驅動開發 skill 同相關嘅指導代理。第七，hooks 不斷啟動：每次編輯都格式化、型別檢查、秘密資料掃描，同埋對多餘除錯日誌嘅警告。第八，審查透過 code-review command 同安全掃描進行。而第九，一個停止 hook 寫低會話摘要，並且抽取任何學到嘅嘢。每一層都有參與。嗰種編排就係成件事嘅重點。

有一個經常令人混淆嘅地方值得搞清楚：ECC 住喺兩個屋企。有原始碼 repo，係一個 git clone，仲有你安裝咗嘅設定，係住喺你各工具嘅設定目錄裏面。原始碼 repo 包含完整嘅目錄、腳本同指南。你安裝咗嘅設定淨係包含你實際裝咗嘅組件。你貢獻或者客製化嗰陣，你改嘅係原始碼，而你調整你實時設定嗰陣，你改嘅係安裝咗嘅設定。原始碼係透過拉取新改動嚟更新，而你安裝咗嘅設定係透過重新執行安裝器或者更新外掛嚟更新。安裝器由原始碼讀取，將解析好嘅檔案寫入你嘅設定目錄，而狀態儲存記住嗰個對應關係，咁 doctor、repair 同解除安裝工具就可以精準咁運作。畀外掛用戶嘅一點提示：用 Claude Code 外掛嗰陣，原始碼係一個受管理嘅快取，而唔係一個你要自己維護嘅 clone。如果你想複製啲 rules，你仲係要另外 clone 個 repo，因為外掛無法分發 rules。

仲有一啲命名你會撞到，可能會令人混淆。ECC 有三個刻意整到唔同嘅公開識別碼。GitHub 原始碼 repo 用一個名。Claude marketplace 外掛用一個簡短嘅識別碼。而 npm 套件就用另一個名。所以安裝 commands 係刻意用唔同嘅名。外掛 id 保持簡短，係為咗滿足嚴格嘅驗證器，而 npm 套件就保留咗佢原本嘅名。比較舊嘅帖文可能會顯示一個比較長嘅舊版 marketplace id，你應該當佢已經棄用。

咁，重點精華。ECC 係四層：身份、目錄、轉接器同運行時，上層管治下層。目錄，由 agents、skills、commands、rules 同 MCP 設定組成，係你會安裝同呼叫嘅嘢。安裝器、清單同狀態儲存，將嗰個通用目錄翻譯落每一款框架。運行時，由 hooks 同腳本組成，係實時強制執行標準嗰個。有兩個屋企：原始碼 repo 同你安裝咗嘅設定。而三個唔同嘅名，全部指嘅都係同一個專案。
