Chapter twelve. Cross-harness use.

Here's the big idea: write once, run everywhere. Despite the name Everything Claude Code, ECC is multi-tool. It calls itself the first plugin to maximize every major AI coding tool. You author your agents, skills, and rules once, and the installer translates them into each harness's native layout.

Two architectural decisions make this work. First, a universal context file at the repo root is read by Claude Code, Cursor, Codex, and OpenCode alike, while GitHub Copilot uses its own instructions file instead. Second, a don't-repeat-yourself adapter pattern lets other harnesses reuse Claude Code's hook scripts instead of duplicating them. Cursor's adapter, for example, transforms Cursor's event data into the shape Claude's hook scripts expect. The install target is just a flag, where you choose Claude, Cursor, Codex, OpenCode, Zed, and so on.

Let's look at the harness scorecard, because feature coverage varies by tool, and these counts are a snapshot that drifts. Claude Code is the richest, with sixty-four agents, eighty-four commands, two hundred and sixty-two skills, eight hook events, thirty-four rules, fourteen MCP connectors, and its own context files. Cursor shares the agents, commands, and skills, has even more hook events at fifteen, carries the rules with its own frontmatter format, and uses the universal context file. Codex, in both its app and command-line form, shares agents and uses an instruction-based approach for commands and rules, has no hook events yet, and uses a configuration file for its handful of MCP servers. OpenCode has its own native set: twelve agents, thirty-five commands, thirty-seven skills, eleven hook events, thirteen rules, and full MCP support. And GitHub Copilot is the lightest, with a handful of prompts, instruction-based skills and rules, no hook events, and a single always-on instructions file. Read it this way: Claude Code is the reference target with the richest surface. Cursor and OpenCode actually have more hook events. And Codex and Copilot lack a hook system, so enforcement there is instruction-based rather than runtime.

Claude Code is the primary, native target. Everything in this book applies directly. You install via the plugin or the manual installer, and it requires the command-line tool at version two-point-one-point-oh or later.

For Cursor, pre-translated configs live in its config directory, and you install by targeting Cursor and your languages. What you get: fifteen hook events, sixteen thin hook scripts that delegate to the shared scripts through an adapter, thirty-four rules in Cursor's frontmatter format, agents installed with a prefix to avoid collisions, plus shared skills, commands, and MCP. Memory isolation matters here. Because Cursor reuses the same hook scripts, ECC automatically keeps Cursor's memory out of your Claude directory, using a session-start hook that sets a separate data home, with a default location, an override file, and an always-on rule. If you deliberately want to share memory with Claude Code, you point the data-home variable at your Claude directory. Cursor's hook system is actually richer than Claude's, with events like before-shell-execution, after-file-edit, before-submit-prompt for secret detection, before-tab-file-read which blocks the tab feature from reading sensitive files like environment and key files, and before-and-after MCP execution for audit logging.

For Codex, the macOS app and command-line tool, there's first-class support through the universal context file plus a configuration file. The cleanest setup is the sync script, which does an add-only merge into your Codex config, with a preview option and a flag to refresh servers. Or you can just run Codex in the repo, since the context file and config directory are auto-detected. The reference config intentionally doesn't pin the model or provider, since Codex uses its own default. What's included: top-level approvals, sandbox, and web-search config, six MCP servers or seven with a database added, thirty-two skills, two profiles, a strict read-only one and a full auto-approve one, and three sample agent roles, an explorer, a reviewer, and a docs researcher. The key limitation: Codex has no hook execution parity yet, so ECC enforcement is instruction-based, through the context file, optional instruction overrides, and sandbox and approval settings. The experimental Codex plugin-marketplace path is fragile upstream, so prefer the sync script.

For OpenCode, there's full plugin support. You just run it in the repo, and it auto-detects its config. Or you install the published plugin from npm and reference it in the OpenCode config. Note that the npm plugin enables ECC's OpenCode hooks, events, and plugin tools, but it does not auto-add the full command, agent, and instruction catalog. For the complete setup, run OpenCode inside the repo, or copy the config assets into your project and wire up the instructions, agents, and commands. OpenCode's plugin system is sophisticated, with more than twenty event types. ECC maps Claude's hooks onto OpenCode events and adds extras like file-edited and language-server diagnostics. It also exposes six native custom tools, like run-tests, check-coverage, and security-audit.

For GitHub Copilot in VS Code, there's no extra tooling. Copilot Chat reads instruction and prompt files automatically. There's an always-injected instructions file with core rules for coding style, security, testing, and git. There are reusable on-demand prompt files for things like plan, test-driven development, security review, build fix, and refactor. And there's a settings file with per-task instruction overlays and prompt-file enablement. The limitation: Copilot has no hook system and no subagent API, so ECC's automations and agent delegation don't apply. But you still get the full coding philosophy in every chat.

There's also a long tail of other tools. There's experimental project-local support for the Gemini command-line tool. There's a conservative Zed adapter, where you keep API keys in Zed's own settings rather than the repo. There's a tightly integrated setup for Antigravity. There are project-local selective-install adapters for JoyCode and CodeBuddy. There's a home-directory adapter for the Qwen command-line tool. And for non-native tools like Grok, there's a manual fallback path. Each of these has its own guide in the docs.

You can also run ECC on a custom endpoint or gateway. ECC doesn't hardcode Anthropic's transport, so it works through Claude Code's normal gateway support. You set a base-URL environment variable and an auth-token variable, then launch. If your gateway remaps model names, you configure that in Claude Code, not ECC, because ECC's skills, hooks, commands, and rules are provider-agnostic once the assistant works. One caution: a malicious project overriding that base-URL variable was a real-world vulnerability, which we'll cover in the security chapter.

So, the key takeaways. ECC is multi-harness: author once, and the installer adapts it, just by changing the target flag. The universal context file is the shared context, and a don't-repeat-yourself adapter reuses hook scripts across tools. Claude Code is the reference, Cursor and OpenCode have more hook events, and Codex and Copilot lack hooks, so enforcement there is instruction-based. Isolate memory between harnesses with the data-home variable. And the setup paths are: a sync script for Codex, run-in-repo or the npm plugin for OpenCode, and instruction and prompt files for Copilot.
