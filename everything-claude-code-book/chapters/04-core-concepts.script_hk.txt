第四章。核心概念：六大建構模塊。

ECC 入面所有嘢，都係由六種嘢組成。學識呢六種，成個系統就會清晰起嚟。呢一章係地圖，而跟住嘅章節會逐個放大嚟睇。

我哋快速睇吓呢六樣嘢。第一，agents。呢啲係你委派工作畀佢嘅、有明確範圍嘅專家。佢哋以個別檔案嘅形式存在，而你互動嘅方式，就係由 orchestrator 委派工作畀佢哋，不過你亦都可以直接要求其中一個。第二，skills，呢啲係主角。佢哋係可重用、多步驟、捆綁住知識嘅工作流程。每一個都住喺自己嘅資料夾。你呼叫佢哋，佢哋會被自動建議，又或者 agents 會重用佢哋。第三，commands。呢啲係觸發一個工作流程嘅斜線項目，而你互動嘅方式，就係打一個斜線同 command 個名。第四，hooks。呢啲係由事件觸發嘅自動化，由一個設定檔對應到細細嘅腳本，而你互動嘅方式，就係你唔使互動。佢哋會自動啟動。第五，rules。呢啲係要永遠遵守嘅指引，有通用嘅，亦有各語言嘅，而佢哋每一次會話都會被載入上下文。而第六，MCP 設定。呢啲係連接外部服務嘅連接器。你逐個專案啟用佢哋，而模型會呼叫佢哋暴露出嚟嘅工具。喺呢六樣之中，skills 係標準嘅介面。當你拿唔定主意，就用 skills 嘅角度去諗。

呢度有一個好用嘅方法，將呢六樣嘢記入腦。Rules 係憲法：被動、永遠存在、約束住一切。Skills 係戰術手冊：耐用嘅「我哋喺呢度係咁做某件事」。Commands 係捷徑：方便、用打字觸發，而且越嚟越只係 skills 之上嘅薄薄包裝。Agents 係團隊：orchestrator 將工作交畀嘅專家，佢哋通常會用到 skills。Hooks 係反射動作：佢哋喺事件發生時啟動，唔使任何人開口。而 MCP 係電話線：模型點樣接觸到 GitHub、一個資料庫，或者一個瀏覽器。一個單一嘅功能請求可能會掂到全部六樣。一個 command 啟動一個 skill，個 skill 委派畀一個 agent，全部受 rules 約束，每次編輯都有 hooks 啟動，仲有一個 MCP 伺服器去抓取資料。嗰種編舞就係 ECC。

我哋放大嚟睇 skills 對 commands 對 hooks，呢三個主動嘅介面。呢三個係人哋最容易混淆嘅，因為三個都可以運行一個工作流程。分別在於觸發方式同耐用程度。一個 skill 係被呼叫嘅，由你或者自動觸發，而且佢係耐用嘅單位。新邏輯就應該放呢度。一個 command 係你打出嚟嘅，而 ECC 保留 commands 係為咗相容，但佢正將佢哋嘅邏輯遷移入 skills。一個 hook 係由事件觸發，例如一次檔案編輯或者一個會話開始，而你永遠唔會打佢出嚟。呢度有一條經驗法則。問吓你自己：呢樣嘢係咪應該喺每次發生某件事嘅時候，無論點都要運行？如果係，咁佢就係一個 hook。又或者問：呢樣嘢係咪一個我揀去運行嘅工作流程？如果係，咁佢就係一個 skill，再加一個可選嘅 command 捷徑。

喺六大模塊之上，仲有兩個橫跨各處嘅能力，將一切連繫埋一齊。佢哋有自己嘅章節，因為佢哋就係令到 ECC 感覺有生命、而唔係靜態嘅嘢。第一個係記憶。Hooks 儲存你嘅會話狀態，並且喺下一次重新載入佢，咁助手就唔使每日由零開始。第二個係持續學習。重複嘅模式變成 instincts，而一個 evolve command 會將佢哋聚集成全新嘅 skills。個目錄係由你自己嘅使用成長出嚟。

仲有一個值得知嘅前言區慣例。Agents 同 skills 其實只係 Markdown 檔案，喺最頂有一個細細嘅結構化標頭。個標頭話畀框架知點樣對待呢個檔案。你會不斷見到呢個模式：一個名、一段關於幾時用佢嘅描述，而 agents 仲會有佢哋應該用嘅工具同模型，至於 skills 就會註明佢哋嘅來源。個描述欄位擔起咗重任。佢就係框架用嚟決定幾時要拎呢個 agent 或者 skill 出嚟嘅依據。含糊嘅描述會被忽略。精準嘅就會被採用。

咁，重點精華。ECC 係六大模塊：agents、skills、commands、hooks、rules 同 MCP 設定。Skills 係標準。Commands 越嚟越只係薄薄嘅捷徑。Hooks 係自動嘅。Rules 約束、agents 執行、MCP 連接，而記憶加學習令到佢隨時間進步。Agents 同 skills 其實只係 Markdown 加一個結構化標頭，而個描述就係驅動自動選取嘅嘢。

而家我哋由你委派工作畀佢嘅團隊開始講：agents。
