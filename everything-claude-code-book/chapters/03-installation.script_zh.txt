第三章。安装和配置。

这一章要做到完全正确，因为一个马虎的安装，是 ECC 让人感觉坏了或者重复了的头号原因。好消息是，这些规则很简单。

我们从一条可以避免大部分问题的规则开始讲。挑恰好一条安装路线，永远不要把它们叠在一起。有两条路线。第一条是插件，是 Claude Code 推荐的默认做法。第二条是手动安装器，或者说是命令行界面。最经典的坏配置，就是执行了插件安装，接着又再执行完整安装器。插件已经加载了 ECC 的 skills、commands 和 hooks。完整安装器接着又把同样的东西复制进你的用户目录，搞到你有重复的 skills 和重复的运行时行为。如果你已经这样做了，你第一站应该是本章稍后的重设部分，在你做任何其他事之前。

现在讲要求。你需要 Claude Code 的命令行工具，2.1.0 版或者之后的版本，因为插件的 hook 自动加载要靠它。你需要 Node，因为 hooks 和脚本是基于 Node 的，可以在 Windows、macOS 和 Linux 跨平台运行。你也需要一个包管理器，无论是 npm、pnpm、yarn 还是 bun。ECC 会自动检测你在用哪个。

我们走一遍路线 A，也就是插件，是 Claude Code 推荐的做法。在一个 Claude Code 会话里面，你加入指向 ECC repo 的 marketplace，然后安装 ecc 这个插件。你也可以在你的 Claude 配置文件里，用声明式的方法接上它，在那里列出 marketplace 并启用插件。这样就立刻给了你 agents、skills、commands 和 hooks。但不包括 rules，这一点我们待会儿会再讲。

这里有一个重要的 hook 陷阱，主要是给贡献者注意。不要在插件的清单文件里加一个 hooks 字段。Claude Code 2.1 版和之后，会按照惯例自动加载一个插件的 hooks 配置，如果你再声明一次，就会引致一个 hooks 文件重复的错误。甚至还有一个回归测试在防着这样东西，正正是因为它之前一次又一次地咬到这个项目。而如果你觉得 hooks 太烦扰，或者你只想要 rules、agents、commands 和核心 skills，你可以跳过插件，改用 minimal 的手动配置文件。Hooks 是刻意不包括在 minimal 配置文件里面的。

现在讲路线 B，手动安装器。你 clone 那个 repo，安装依赖，然后用一个配置文件和一个目标去执行安装器。例如，你可能会安装 minimal 配置文件，目标是 Claude Code。在 Windows 上，有一个 PowerShell 版本，而你甚至可以不用 clone，用 npx 安装器来执行它。

我们讲一下配置文件。Minimal 配置文件给你 rules、agents、commands 和核心工作流程 skills，但没有 hooks 运行时。Core 配置文件给你标准那一套，而你可以保留 core 的同时，用一个标志停用 hooks。Full 配置文件给你所有东西，而你应该只在完全手动的路线上用它，永远不要在插件安装之后用。

ECC 还有一个由清单驱动的选择性安装器，所以你可以精准地加入或者移除组件。你可以稍后再加回只有 hooks 运行时。你可以保留 core 但跳过 hooks。或者你可以在 minimal 之上，加入一个特定能力，例如机器学习。而如果你不知道要装什么，问顾问吧。有一个 consult command，你描述一个需求，例如安全审查，或者模型训练和部署，它就会交回相符的组件、相关的配置文件，和预览和安装的 commands。如果你想检查确切的文件计划，你可以在安装之前先预览。

这里有样东西让人人都绊到：rules 永远需要手动复制。Claude Code 插件无法分发 rules。这是上游的限制。所以无论你挑了哪条路线，如果你想要 ECC 那些要永远遵守的指引，你都要自己复制它。你可以在用户级别安装它们，应用到所有项目，或者在项目级别安装，只应用到一个项目。几条经验法则。从 common 的 rules 加一个你实际会用的语言包开始。复制整个语言目录，而不是里面单个的文件，这样相对引用和文件名才不会撞。而除非你真的想加载这么多上下文，否则不要复制每一个 rules 目录。

如果你走的是手动路线，正确地安装 hooks。不要亲手把 repo 的 hooks 配置复制进你的配置文件。那个文件是面向插件和 repo 的，不支持原样复制。改用安装器吧，这样那些 command 路径才会被正确地改写。安装器把解析好的 hooks 写入合适的位置，并且不动你的配置文件。而如果你用了插件，你已经自动加载了 hooks，所以不要再把它们复制进你的配置。

现在讲重设和卸载。如果 ECC 让人感觉重复、烦扰或者坏了，不要在它自己上面再装一次。先诊断和清理。从生命周期包装器开始。你可以列出装了什么、执行一个 doctor command 去诊断、执行一个 repair command 去还原 ECC 管理的文件，以及用试运行模式执行卸载去预览。还有一个带预览选项的直接卸载脚本。ECC 只移除它在安装状态里记录下来的文件，所以它不会删除无关的文件。如果你叠用了几种方法，就按这个顺序清理。第一，移除 Claude Code 插件安装。第二，在 repo 根目录执行 ECC 卸载，这样就会移除状态管理的文件。第三，删除任何你手动复制过的额外 rule 文件夹。而第四，只用一条路线，重新安装一次。一个安心的保证：如果你本机的 Claude 配置被清掉了，你不需要再买任何东西。在重新安装之前，只执行 list-installed，接着 doctor，再接着 repair 就行。

讲一讲 MCP 服务器：它们是可选择加入的。插件安装刻意不自动启用 ECC 内置的 MCP 服务器定义。这样可以在严格的网关上避免过长的工具名，并且保护你的上下文窗口。ECC 恰好内置一个默认连接器，给 Chrome 开发者工具用。其余一切，要么是一个包住命令行工具或者 REST API 的 skill，要么就是一个可选择加入的目录项目。要加入 MCP 服务器，你可以用 Claude Code 的 MCP command 做实时改动，或者把你想要的服务器复制进一个项目配置文件，给 repo 本地访问。如果你已经在跑自己的那份，你可以设置一个环境变量，这样安装和同步流程就会跳过它们。而永远记得把任何占位密钥换成真正的。上下文窗口的纪律，后面会有自己的一章。

还有一个配置提示：多模型 commands 需要额外配置。Multi-plan、multi-execute、multi-backend、multi-frontend 和 multi-workflow 这些 commands，不是基本安装涵盖的。它们要靠一个独立的工作流程运行时，你用它自己的 npx command 来安装。那个提供了这些 commands 预期的外部依赖。没有它，那些 multi commands 就不会正确地运行。

要验证你的安装，你可以在 Claude Code 里面列出插件提供了什么、从 repo 执行测试套件，或者拍一张操作者就绪度快照，写下一个可携带的交接文件。当你执行 list-installed command 的时候，一个成功的手动安装会显示 ECC 管理的文件。

最后，给其他框架的几个快速指引。完整的跨框架细节后面再讲，但目标标志是关键。你可以把目标设成 Cursor、用 minimal 配置文件把目标设成 Zed、执行一个同步脚本去合并进 Codex，或者干脆启动 OpenCode，它会自动检测它自己的配置目录。

那么，重点精华。只有一条路线：插件或者手动安装器，永远不要两样都用。插件最快，但 rules 仍然需要手动复制，也就是 common 的 rules 加一个语言包。配置文件有 minimal，没有 hooks；core；和 full，而 full 只在手动那条路才用。如果你不确定要装什么，用 consult command。用 list-installed、doctor 和 repair 来重设，接着试运行卸载，然后再重新安装。而 MCP 服务器和 multi commands 是刻意设成可选择加入的。

装好了 ECC，我们就来学一下词汇：六大构建模块。
