第八章。挂钩，Hooks。

我们从 hooks 是什么开始讲。一个 hook 是一个在特定事件启动的自动化：一次工具调用、一个会话开始、上下文快要被压缩。和 skills 和 commands 不同，你永远不会调用一个 hook。是框架触发它的。Hooks 是 ECC 的反射动作：那些应该每次都自动发生、不用你记得去做的格式化、把守、机密信息扫描，和记忆保存。在 ECC 里面，一个配置文件声明事件和匹配器，而每一个项目都会执行一个小小的 Node 脚本。它们用 Node 而不是内嵌 shell，是为了跨平台的可靠性，这是 1.8 版引入的一个刻意的加固。

现在我们走一遍事件生命周期。主要的事件用 Claude Code 的名字，而其他框架就对应回这些。有会话开始，在一个新会话开始时启动，而 ECC 用它来加载之前的上下文、检测你的包管理器，和注入一份记忆摘要。有工具使用前，在一个工具运行之前启动，而 ECC 用它来提醒你用终端多路复用器、把守一次 git push、扫描机密信息、保护配置文件，以及在第一次编辑之前强制做一次事实核查。有工具使用后，在一个工具完成之后启动，而 ECC 用它来格式化代码、类型检查、执行质量关卡、对多余调试日志作出警告，和更新度量。有一个工具使用失败事件，在一个工具出错时为诊断捕捉失败的上下文。有压缩前，在上下文压缩之前启动，并且先把重要的状态保存到一个文件。还有停止，在一个会话结束时启动，写下一份会话摘要、批量格式化和类型检查，并且抽取学到的东西。


有几个有代表性的 ECC hooks，你会见到它们的名字。有一个合并了的 shell 预检，捆绑了质量、多路复用器、push 和关卡的检查。有一个关卡守卫事实强制 hook，会阻止每个文件的第一次编辑，并且要求调查，在允许改动之前问：谁在引入这个、结构是什么，和用户实际上问了什么。有一个配置保护 hook，会阻止对 linter 和 formatter 配置的编辑，引导 agent 去修代码，而不是削弱配置。有一个质量关卡 hook，在编辑之后运行。有一个 console 警告 hook，对付多余的调试日志。有一个上下文监控 hook，会注入关于上下文耗尽、高成本、范围蔓延，或者工具循环的警告。有一个会话开始引导，负责记忆和包管理器检测。还有压缩和停止的 hooks，负责记忆留存。配置里面真正的 command 字符串看起来很吓人，因为每一个都要在很多可能的安装位置之中，先引导到正确的插件根目录，然后才委派给真正的脚本。你永远不用亲手写那些。安装器会生成它。

这里是 ECC 其中一个最好的功能：你可以不用编辑任何文件就调校 hooks。你用环境变量去控制 hook 的行为。有一个严格度配置，可以是 minimal、standard 或者 strict，默认是 standard。你可以用 hook 的 id，以逗号分隔的清单，去停用特定的 hooks。你可以为低上下文或者本机模型，封顶甚至关闭会话开始的上下文注入。你可以设置一个以天计的会话保留窗口。你可以保留上下文、范围和循环的警告，同时隐藏成本估算。而你可以选择加入治理事件捕捉。每一个 hook 都标注着它在哪些配置文件下运行，所以 minimal 配置文件会悄悄地丢掉那些比较吵的，而 strict 就会把所有东西都打开。

现在，一个关于 hooks 文件重复陷阱的警告。这是最多人汇报的 hook 问题。Claude Code 2.1 版和之后，会按照惯例自动加载一个插件的 hooks 配置。如果你又在插件清单里声明 hooks，或者在插件安装之后把 hooks 配置复制进你的配置文件，你就会得到一个 hooks 文件重复的错误。这里是怎样避免它。贡献者永远不应该在插件清单里加一个 hooks 字段，而一个回归测试会强制执行这一点。插件用户不应该把 hooks 复制进他们的配置文件，因为它们已经加载了。而手动用户应该通过安装器、用 hooks-runtime 模块来安装 hooks，这样就会把它们写入合适的位置，并且不动配置文件。


我们看一下一个 hook 的概念形状。如果你剥走那个引导包装，一个 hook 项目其实只是几个字段。有一个匹配器，讲这个适用于哪些工具或者事件，例如一次 shell 调用、一次编辑，或者所有东西。有个 command，是要执行的脚本。有异步和超时的设置，这样这个 hook 就不会阻塞 agent，而它的运行时间有封顶。还有一个 id，是你停用 hooks 时用的把手。还有一个退出码惯例：只在阻挡是刻意的时候才用一个失败码退出，否则就干净地退出，而那些消息应该永远都可以采取行动。指南里面一个最小的手写示例，是一个匹配 JavaScript 或者 TypeScript 文件编辑的 hook，grep 一句多余的调试日志，然后打印一个提醒叫你移除它。

推荐用简单的方法创建 hooks，因为亲手写 hook 配置很麻烦。短篇指南推荐用 hookify 的做法：执行 hookify command，用对话方式描述你想要什么，它就会帮你生成那个 hook。ECC 还内置了几个配套 commands，去列出、配置，和获取 hookify 的协助。如果你真的亲手写一个，就用提供的脚本去验证它。

那实际上 hooks 给了你什么呢？在 standard 配置文件开着 hooks，一个普通的编辑会话会悄悄地得到：编辑之后自动格式化和类型检查，这样就不会再忘记执行 formatter；在有风险的工具调用之前和提示提交之前检测机密信息；一道 git-push 审查关卡，这样那些 push 就不会无声无息；一个用终端多路复用器去运行长时间 commands 的提醒；跨会话的记忆留存；喂给 instincts 的持续学习捕捉；和一个在你撞墙之前警告你的上下文监控。这就是在零持续努力的情况下，得到很多资深工程师的纪律。

那么，重点精华。Hooks 是在工具和会话事件上的自动反射动作，把一个配置文件对应到 Node 脚本。生命周期从会话开始，到工具使用前、工具使用后、压缩前，再到停止。你在运行时用 hook 配置和停用 hooks 的环境变量去调校它们，不用编辑文件。永远不要双重加载 hooks，也就是那个 hooks 文件重复陷阱，并且用对的方法安装它们。而且用 hookify command 去用描述创建 hooks，而不是亲手写配置。
