第十一章。日常工作流程。

呢一章你會不斷返嚟睇。佢將所有組件變成具體、可重複嘅回圈。複製佢哋、改編佢哋，佢哋就會變成肌肉記憶。

我哋由主迴圈開始：規劃，然後測試驅動開發，然後審查，然後驗證，然後學習。差唔多每一次 ECC 會話都遵循同樣嘅形狀。第一步係規劃。你用好似「加入 OAuth 登入」嘅嘢去執行 plan command。Planner 代理產出一個計劃，而你確認佢。第二步係測試驅動開發。你呼叫測試驅動開發工作流程 skill，行紅、綠、重構，由測試驅動指導帶領。第三步係審查。你執行 code-review 同 security-scan commands，佢哋會呼叫 code reviewer 同 security reviewer。第四步係驗證。你執行品質關卡、修任何建構錯誤，同確認測試覆蓋率至少八成。而第五步係學習。你執行 learn-and-evaluate command 並且儲存個會話，產出一個 instinct 同一份會話摘要，仲有停止 hook 幫手。圍住成件事，hooks 無形咁格式化、型別檢查同掃描秘密資料，而 rules 約束每一步。喺各階段之間，清除上下文並且將中途嘅輸出存做檔案，咁你就唔會淹冇你嘅上下文視窗。

呢度係開始一個新功能嘅食譜。你用好似「用 OAuth 加入用戶驗證」嘅描述去執行 plan command。Planner 產出一個分階段嘅實作藍圖，而你批准佢。然後你呼叫測試驅動開發工作流程：測試驅動指導先寫失敗嘅測試，然後寫最小嘅實作。然後你執行 code-review，而 code reviewer 檢查個差異，如果乾淨就交返零個發現。點解係呢個順序？先規劃後執行，阻止範圍亂咁擴散。測試優先，意味住個實作係由一個規格帶領。而即刻審查，就趁上下文仲新鮮時捉到問題。

呢度係修一個 bug 嘅食譜。你呼叫測試驅動開發工作流程，而測試驅動指導寫一個重現個 bug 嘅失敗測試，一個有效嘅紅色狀態。然後你實作個修復並且確認個測試轉綠。然後你執行 code-review 去捉迴歸。呢度嘅紀律係之前嗰個紅色關卡：喺你郁生產程式碼之前，你必須有一個真係因為預期嘅原因而失敗嘅測試。冇重現，就唔修。

呢度係為生產做準備嘅食譜。你執行一個安全掃描，喺嗰度 security reviewer 做一個跟 OWASP 標準對齊嘅審計，又或者用 AgentShield。你呼叫端對端測試 skill，喺嗰度端對端執行員喺一個真實嘅瀏覽器裏面演練關鍵嘅用戶流程。你執行 test-coverage command 去驗證八成以上嘅覆蓋率。而你執行品質關卡，做一個對照專案標準嘅最後檢查。

呢度係用 sub-agents 編排一個更大型建構嘅食譜。對於多階段嘅工作，倚靠依次序階段嘅編排。階段一係研究，產出一份研究摘要，用一個平嘅模型。階段二係規劃，產出一個計劃，用一個強大嘅模型去做困難嘅設計。階段三係實作，產出程式碼同測試，喺一個平衡嘅模型上面。階段四係審查，產出一份審查。而階段五係驗證，佢要麼完成，要麼回圈返去失敗嗰個階段。編排 commands 將呢啲模式打包，畀你建立一個 MVP、加入一個功能、修一個缺陷等等。而對於多模型協作，有啲 multi commands，不過記住佢哋需要裝咗佢哋獨立嘅工作流程運行時。

我哋講吓並行化：一次過做多啲。當一個會話唔夠用，ECC 嘅指引係刻意嘅，唔係淨係開十個終端機。第一，為唔重疊嘅工作分叉。用一個 fork command 去為問題或者研究分叉一段對話，而你嘅主對話繼續編輯程式碼。將程式碼改動留喺主對話，用分叉去做唯讀嘅探索。第二，為重疊嘅程式碼工作用 git worktrees。當多個實例必須編輯重疊嘅程式碼，就畀每一個佢自己嘅簽出，並且每個 worktree 行一個實例，為你嘅對話改名，咁你就唔會跟唔到。第三，有級聯方法：喺右邊嘅分頁開新任務，由左掃到右，由最舊掃到最新，而且一次最多聚焦喺三四個任務。指導原則係呢條問題：用最低可行嘅並行量，你可以做到幾多？淨係喺真正必要嘅時候先加一個終端機。

呢度有一個畀全新專案嘅好模式：雙實例專案啟動，喺一個空 repo 上面用兩個助手實例。第一個實例係一個搭建代理，鋪低專案結構同設定。第二個實例係一個深度研究代理，連接到服務同網絡搜尋、寫產品需求文件、畫架構圖，同收集真實嘅文件。然後佢哋匯合：個骨架遇上嗰個經過研究嘅計劃。

跟住係日終交接。唔好等一個會話蒸發。捕捉佢。執行 learn-and-evaluate command 去抽取可重用嘅模式，並且喺儲存之前自我評估佢哋，然後儲存個會話去為狀態影快照。又或者倚靠停止 hook 去自動寫一份摘要。第二朝早，恢復個會話去重新載入最新嗰個並且繼續。而對於一個可攜帶、可分享嘅狀態，好適合站立會議或者交接畀隊友，有一個 status command 寫一個 markdown 交接檔案。

ECC 將驗證當做一個一等嘅工作流程，而唔係事後孔明。有兩種評估風格。基於檢查點嘅，喺嗰度你設定明確嘅檢查點、對照準則驗證，並且喺繼續之前修復。同持續嘅，喺嗰度你每隔幾分鐘或者喺重大改動之後運行，用完整嘅測試套件同 linter。仲有兩種解讀可靠性嘅方式。第一種係 pass-at-k，喺嗰度 k 次嘗試之中至少一次成功，你喺淨係需要佢行得到嗰陣用佢。第二種係 pass-to-the-k，喺嗰度 k 次嘗試全部成功，你喺一致性係必要嗰陣用佢。評估框架同驗證回圈 skills 實作咗呢啲，而品質關卡就係日常嘅關卡。

由指南嚟嘅幾個高槓桿習慣，細細嘅嘢但回報大。停用自動壓縮，並且喺合乎邏輯嘅斷點策略性咁壓縮。喺一個終端機多工器裏面運行長 commands，咁你就可以分離、重新連接，同串流日誌。ECC 甚至會提醒你。用語意搜尋代替純文字搜尋，將搜尋 token 減半。設定一條狀態列，顯示你嘅分支、上下文百分比、模型，同待辦事項。逐個專案保持 MCP 精簡。並且喺 commit 之前審查差異，等 git-push hook 把守你。

咁，重點精華。主迴圈係規劃、測試驅動開發、審查、驗證、學習，全部包喺 hooks 同 rules 入面。喺修復之前，用一個有效嘅失敗測試重現 bug。將大型工作編排成依次序嘅階段，喺各 agents 之間傳遞檔案。用分叉、git worktrees 同級聯方法去並行化，但要最低限度。捕捉每一個會話，咁聽日就贏在起跑線。而且用檢查點同持續評估，加 pass-at-k 對 pass-to-the-k 嘅思維去驗證。
