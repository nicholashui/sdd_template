第七章。指令，Commands。

我哋講吓 commands 係乜，以及佢哋去緊邊度。一個 command 係你打入你框架嘅斜線項目，例如 plan、code-review，或者 build-fix。每一個對應到一個 Markdown 檔案，並且觸發一個工作流程。

但呢度有個重要嘅定位：commands 係一個相容層。ECC 刻意將耐用嘅邏輯遷移入 skills，並且喺嗰個遷移過程中，將 commands 目錄當做受維護嘅斜線項目相容。已退役嘅短名，例如舊嘅測試驅動開發同評估 commands，已經搬入一個 legacy-shims 資料夾，淨係畀人明確選擇加入。

咁點解 commands 仍然存在呢？因為打一個短 command，比描述成個工作流程更快、更易被發現，亦因為跨框架嘅對等有時需要一個有名嘅項目。將 commands 諗成真正引擎之上嘅符合人體工學嘅捷徑，而真正嘅引擎就係 skills。一個命名提示：外掛安裝用標準嘅命名空間形式，帶一個 ecc 前綴，而手動安裝就保留比較短嘅斜線形式。兩者都去到同樣嘅邏輯。

我哋過一過 command 嘅速查表。Repo 內附一個快速參考檔案，而啲 commands 按工作分組。確實有邊啲可用，視乎你嘅安裝同框架，所以打一個斜線去睇吓有啲乜實時可用。

喺核心工作流程，有 plan，佢會重述需求、評估風險，並且寫一個逐步嘅計劃，然後喺郁程式碼之前等你確認。有 code-review，係對已改動檔案嘅一次完整品質、安全同可維護性審查。有 build-fix，佢會偵測同修復建構錯誤，並且委派畀啱嘅 resolver。有 quality-gate，佢會根據專案標準執行驗證關卡。仲有 refactor-clean，佢會移除死碼並且整合重複嘅部分。

至於測試，有一個 test-coverage command，再加 Go、Rust、Kotlin、C++、React 同 Flutter 嘅特定語言測試項目。通用嘅測試驅動開發 command 而家住喺 legacy shims，而你應該偏向用測試驅動開發 skill。

至於程式碼審查，有畀 Python、Go、Rust、Kotlin、C++、React、FastAPI 同 Flutter 嘅各語言 commands。至於建構修復，有畀 Go、Rust、Kotlin、C++、Gradle、React 同 Flutter 嘅各語言 commands。

至於規劃同多代理工作，有 plan、一個產品需求計劃、multi-plan、multi-execute、multi-backend、multi-frontend 同 multi-workflow 呢啲 commands，同一個編排家族，負責建立一個 MVP、加入一個功能、修一個缺陷、改一個功能，同精修程式碼。

至於會話同上下文，有 save-session、resume-session、一個 sessions 清單、checkpoint、一個 aside command，畀你快速問一條題外問題又唔會失去你嘅任務，同 context-budget。

至於學習同改進，有 learn、learn-eval、evolve、promote、instinct status、instinct import 同 export、skill-create、skill-health、rules-distill、prune，同 projects。

至於文件同研究，有一個文件查詢 command、一個 update-docs command，同一個 update-codemaps command。至於回圈同自動化，有 loop-start、loop-status、一個 REPL command，同一個長時間運行嘅回圈 command。至於框架同基礎設施，有負責框架審計、模型路由、程序管理、成本報告、自動更新、將行為變成 hooks，同專案初始化嘅 commands。而至於拉取請求工作流程，有負責開一個 PR、審查一個 PR、分析 PR 測試嘅 commands，同一個由產品需求一路行經規劃、實作、commit 到 PR 嘅家族。

呢度有一個由快速參考整理出嚟嘅快速決策指南，一個好用嘅心智流程圖。開始一個新功能？先執行 plan，然後測試驅動開發 skill。啱啱寫完程式碼？執行 code-review。建構壞咗？執行 build-fix。需要實時文件？用一個函式庫名執行文件 command。會話就嚟完？執行 save-session，或者 learn-eval command。第二日恢復？執行 resume-session。上下文越嚟越重？執行 context-budget，然後 checkpoint。想捕捉你學到嘅嘢？執行 learn-eval，然後 evolve。而執行緊重複嘅任務？執行 loop-start。

快速講一講 legacy shims 同改咗啲乜。有啲舊愛將，被退役做獨立嘅短 commands，而家會指你去返嗰個 skill。舊嘅測試驅動開發 command 指向測試驅動開發 skill。舊嘅端對端 command 指向端對端測試 skill。舊嘅評估 command 指向評估框架 skill。舊嘅 verify command 指向驗證回圈 skill。而舊嘅 orchestrate command 指向多工工作流程 skill 或者 multi-workflow command。呢啲住喺 legacy-shims 資料夾，係可選擇加入嘅。如果你嘅肌肉記憶想要返個舊名，就複製嗰一個單一嘅墊片，但要知道嗰個 skill 先係真正嘅嘢。

咁寫一個 command 又點呢？Commands 係最簡單嘅模塊：一個 Markdown 檔案，標頭有一段描述，正文有提示同指令。但按照 ECC 嘅政策，淨係喺真係需要一個墊片嗰陣先加入或者更新一個 command，亦即係為咗遷移或者跨框架對等。否則，將邏輯放入一個 skill。然後用提供嘅腳本去驗證佢。

咁，重點精華。Commands 係打出嚟嘅斜線捷徑，而因為 ECC 係技能優先，佢哋係一個相容層。外掛形式用一個 ecc 前綴，手動形式係淨係淨咁嘅斜線，而兩者都打中同一個引擎。快速參考檔案將 commands 按工作分組，而你可以打一個斜線去睇吓裝咗啲乜。已退役嘅短名住喺 legacy-shims 資料夾，並且指返去 skills。而你應該偏向建立 skills，多過建立新嘅 commands。
