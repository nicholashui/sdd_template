第十四章。Token 優化同效能。

代理式編程可以好快變得昂貴，而且當你填滿上下文視窗，品質其實會下降。ECC 將成本同上下文當做有真實槓桿嘅工程問題。呢一章將佢哋收集埋一齊。

我哋由真正有作用嘅設定開始講。喺你嘅 Claude 設定檔裏面，有幾個選擇最重要。模型設定預設係最強大嘅模型，但建議係用平衡嘅模型。咁可以削減大約六成成本，並且應付超過八成嘅編程任務。最大思考 token 設定預設接近三萬二千，但將佢降到一萬，就削減咗每個請求大約七成嘅隱藏思考成本。自動壓縮百分比預設係九十五，但將佢設做五十，就會更早壓縮，喺長會話裏面畀到更好嘅品質。而訂閱用戶可以關閉成本警告，咁就隱藏咗 API 費率嘅估算，同時保留上下文、範圍同回圈嘅警告。淨係喺你需要深度嗰陣，先升級去最強大嘅模型。

唯一最大嘅成本槓桿，係模型路由：令模型對應到任務。換句話講，就係唔好咩都用最強大嘅模型。呢度係 ECC 推薦嘅路由。至於探索同搜尋，用最平嘅模型，因為佢快、平，而且搵檔案綽綽有餘。至於簡單嘅單檔案編輯，又係最平嘅模型，因為指令清晰而風險低。至於多檔案實作，平衡嘅模型，係編程嘅最佳平衡。至於拉取請求審查，平衡嘅模型，可以負擔得起咁捉到細微之處。至於複雜嘅架構，最強大嘅模型，做深度推理。至於安全分析，最強大嘅模型，因為你輸唔起漏嘢。至於除錯困難嘅 bug，最強大嘅模型，因為你必須將成個系統記住喺腦。而至於寫文件，最平嘅模型，因為結構簡單。重點：大約九成嘅編程，預設用平衡嘅模型。淨係喺第一次嘗試失敗、任務橫跨五個或者以上檔案、係一個架構決定，或者係安全關鍵嗰陣，先升級去最強大嘅。呢個亦都係點解 sub-agents 咁重要：一個 sub-agent 架構，等你可以為每一個子任務指派最平但足夠嘅模型，而唔使為一次檔案搜尋付頂級費率。ECC 嘅 model-route command 同 harness-optimizer agent，幫你將呢樣嘢自動化。

跟住，保護上下文視窗，因為喺接近上限嗰度，品質會跌落懸崖。用幾個方法守住佢。為大型重構同多檔案功能，避開視窗最後嘅大約兩成，不過單次編輯同文件可以容忍更高嘅使用率。保持 MCP 精簡，啟用少過十個伺服器同少過八十個工具，單係咁就可以收回超過十萬個 token。封頂會話開始嘅上下文，又或者為本機、低上下文模型完全關閉佢。用策略性壓縮：停用自動壓縮，並且喺合乎邏輯嘅斷點壓縮。ECC 有一個推你一把嘅 hook，而壓縮前 hook 會先儲存狀態。用 context-budget command 去睇 token 去咗邊並且削減開銷。並且倚靠上下文監察，即係嗰個喺耗盡、失控成本、範圍蔓延同工具回圈警告你嘅 hook。

仲有工具層級嘅 token 節省。用語意搜尋代替純文字搜尋。喺 ECC 嘅五十項任務基準測試裏面，配合 Claude Code 嘅語意搜尋，喺相似或者更好嘅評定品質下面，用咗大約一半嘅 token。保持程式碼庫模組化，檔案係數百行而唔係數千行，咁讀取更平，亦更常一擊即中，而呢個亦都係一條 rule。用一個包住命令列工具嘅 skill，代替一個成日存在嘅 MCP，咁可以將工具描述擋喺上下文之外。並且用 codemaps，等 skills 攜帶輕量嘅地圖，咁助手就唔使重新探索咁去導航。

至於並行化，即係日常工作流程嗰章講過嘅，總結係咁：並行工作令吞吐量倍增，但 ECC 建議最低可行嘅量。為唔重疊、唯讀嘅工作用分叉，好似問題同研究，而主對話編輯程式碼。為重疊嘅程式碼工作用 git worktrees，每個簽出一個實例，並且為你嘅對話重新命名。並且用級聯方法，最多聚焦喺三四個任務，由最舊去到最新。反對淨係因為你做得到就開十個終端機嘅指引係：每一個額外嘅實例，都係你要管理嘅上下文。淨係喺必要時先加一個。

仲有驗證效率。因為你冇驗證而要重做工作，本身就係一種成本。ECC 嘅評估定位：喺合乎邏輯嘅里程碑運行檢查點評估，並且喺重大改動之後運行持續評估。喺你淨係需要佢行得到嗰陣用 pass-at-k，喺你需要一致性嗰陣用 pass-to-the-k。評估框架同驗證回圈 skills，加品質關卡，實作咗呢樣嘢。早啲捉到一個迴歸，比佢出貨之後先捉到平好多。

至於成本可見度，有一個 cost-report command 畀一份成本摘要，同 context-budget command 做 token 開銷分析。儀表板同 status 工具拎出度量，咁成本就唔再係一個謎，而一個 metrics-bridge hook 為狀態列同上下文監察保持一個運行中嘅會話聚合。

如果你想要一個單一、又快又平嘅起步設定嚟開始，呢度用文字講出嚟。喺你嘅設定，用平衡嘅模型、將最大思考 token 設做一萬，並且將自動壓縮設做五成。然後將你嘅 hook 設定檔設做 standard、封頂會話開始嘅上下文、逐個專案保持啟用少過十個 MCP 同少過八十個工具、預設用平衡嘅模型而淨係喺真正需要時先升級去最強大嘅、用語意搜尋、保持檔案模組化，並且喺合乎邏輯嘅斷點壓縮。

咁，重點精華。設定平衡嘅模型、一萬個思考 token，同五成自動壓縮，去大幅削減成本。按任務路由模型：搜尋用最平嘅、平衡嘅做預設、最強大嘅淨係用喺困難或者關鍵嘅工作。保護上下文：少過十個 MCP 同八十個工具、封頂會話開始、策略性咁壓縮，並且留意 context-budget。語意搜尋同模組化檔案，實質咁減少 token 使用。並且最低限度咁並行化，同時早啲驗證，用檢查點同持續評估同 pass-at-k 對 pass-to-the-k 嘅思維。
