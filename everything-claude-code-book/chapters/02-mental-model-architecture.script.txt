Chapter two. The mental model and architecture.

The fastest way to understand ECC, rather than just memorize it, is to see it as four stacked layers. Each layer answers a different question, and each layer is shaped by the one above it.

Layer one is identity and governance. This is who ECC is. Three small root files set the constitution. There's a soul file, which holds the core identity and the five principles. There's a rules file, which holds the hard must-always and must-never constraints, plus the required formats for agents, skills, and hooks. And there are the instruction files that the assistant reads, including the agent roster, the security guidelines, and the workflow. This layer is why everything else behaves the way it does.

Layer two is the component catalog. This is the reusable stuff, the heart of the repository, the things you install and invoke. It's organized into agents, skills, commands, rules, and MCP configs. This is the layer most of the book is about.

Layer three is the harness adapters and installer. This is how it lands on disk. The catalog is generic, but each AI tool wants its config in a different place and a different format, so this layer translates. There are install entry points for different platforms. There's a selective-install engine driven by manifests and install scripts. A state store records what got installed, which enables clean updates and uninstalls. And the targets are the per-tool config directories, for Claude, Cursor, Codex, OpenCode, Gemini, Zed, GitHub, and others.

Layer four is the runtime and enforcement. This is what happens while you work. When a session is actually running, this layer is alive. A hooks configuration maps events to small scripts. Those hooks format code, scan for secrets, gate quality, persist memory, and capture learning signals. And metrics and observability feed the operator command-line interface and the ECC two-point-oh control plane.

Here's a mental shortcut for all of that. Identity governs the catalog. The catalog is shaped by the adapters. The adapters land it on disk. And the runtime enforces it. Top governs bottom.

Now let's take a tour of the repository map. You will not touch most of this directly, but knowing the lay of the land removes a lot of mystery. At the root you have the identity and governance files, plus a version file. Then come the catalog directories: agents, with sixty-four specialized sub-agents; skills, with more than two hundred and sixty, each in its own folder; commands, with eighty-four slash commands that stay legacy-compatible but are skills-first; an opt-in archive of retired short command names; rules, with common rules plus per-language packs; hooks; MCP configs; and contexts, which inject content into the system prompt for development, review, and research modes.

There's also a scripts directory holding cross-platform plumbing: the actual hook implementations, shared utilities, the operator command-line interface, and the selective-install planner and applier. There's a Rust control-plane prototype for ECC two-point-oh. There's a desktop graphical interface. There are installer entry points, install manifests, JSON schemas for validation, a test suite, and example configs for things like SaaS, Go, Django, and Rust projects.

Three guides in the repository are pure gold. The repo says it plainly: this repo is the raw code only, and the guides explain everything. The shortform guide covers the foundations, skills, hooks, subagents, MCPs, and plugins, and you should read it first. The longform guide covers token economics, memory persistence, evaluations, and parallelization. And the security guide covers attack vectors, sandboxing, real-world vulnerabilities, and AgentShield. This book distills and organizes all three, but the originals are excellent.

Let's trace how a single request flows through ECC. Say you run the plan command with the request, add OAuth login. Here's what actually happens. First, governance loads. The rules and instruction files are already in context, so the assistant knows the standards: test-driven development, immutability, security. Second, if installed, a session-start hook has already injected memory from prior sessions and detected your package manager. Third, the command or skill fires. The plan command maps to planning logic, which may delegate to the planner agent. Fourth, the agent works in a scoped context, with limited tools and an appropriate model, and returns a plan. Fifth, you confirm, because plan-before-execute means it waits for you. Sixth, implementation proceeds, likely through the test-driven-development skill and the associated guide agent. Seventh, hooks fire continuously: formatting on every edit, type-checking, secret scanning, and warnings about stray debug logs. Eighth, review happens through the code-review command and the security scan. And ninth, a stop hook writes a session summary and extracts any learnings. Every layer participated. That orchestration is the whole point.

There's one frequent source of confusion worth clearing up: ECC lives in two homes. There's the source repository, which is a git clone, and there's your installed config, which lives in your per-tool config directories. The source repo contains the full catalog, scripts, and guides. Your installed config contains only the components you actually installed. You edit the source when you're contributing or customizing, and you edit the installed config when you're tweaking your live setup. The source is updated by pulling new changes, and your installed config is updated by re-running the installer or updating the plugin. The installer reads from the source and writes resolved files into your config directories, and the state store remembers the mapping so the doctor, repair, and uninstall tools can work precisely. One note for plugin users: with the Claude Code plugin, the source is a managed cache rather than a clone you maintain. You still clone the repository separately if you want to copy the rules, because plugins can't distribute rules.

There's also some naming you'll run into that can be confusing. ECC has three public identifiers that are intentionally different. The GitHub source repo uses one name. The Claude marketplace plugin uses a short identifier. And the npm package uses yet another name. So the install commands use different names on purpose. The plugin id is kept short to satisfy strict validators, while the npm package kept its original name. Older posts may show a longer legacy marketplace id, and you should treat that as deprecated.

So, the key takeaways. ECC is four layers: identity, catalog, adapters, and runtime, with the top governing the bottom. The catalog, made of agents, skills, commands, rules, and MCP configs, is what you install and invoke. The installer, manifests, and state store translate that generic catalog onto each harness. The runtime, made of hooks and scripts, is what enforces standards live. There are two homes: the source repository and your installed config. And three different names all refer to the same project.
