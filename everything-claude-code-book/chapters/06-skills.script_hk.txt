第六章。技能，Skills。

我哋講吓點解 skills 係 ECC 嘅核心。如果 agents 係「邊個做嘢」，咁 skills 就係「件嘢點樣做」。一個 skill 係一個可重用、自成一體嘅工作流程：啲步驟、結構、範例、陷阱，有時仲有支援檔案同 codemaps。一個 codemap 係一份程式碼庫嘅輕量地圖，等助手可以唔使燒 token 重新探索就導航到。

ECC 好強調，skills 係標準嘅工作流程介面。新嘅工作流程開發應該先落喺 skills。Commands 仍然存在係為咗相容，但耐用嘅邏輯就屬於 skills。原因要追溯返去之前嗰套複利工作流程嘅理念：一個 skill 會被永遠重用，跨專案同跨會話，而且隨住模型改進，佢會變得越嚟越好。

一個 skill 有三種被使用嘅方式。第一，你用個名去呼叫佢。第二，當佢嘅描述同你正喺度做嘅嘢相符，佢會被自動建議。而第三，一個 agent 會重用佢：一個可以執行你部分 skills 嘅 sub-agent，可以被交一個任務，然後自主咁用嗰啲 skills。

而家我哋睇吓一個 skill 檔案嘅解剖。每一個 skill 住喺自己嘅目錄，入面有一個主 skill 檔案。佢個標頭好細。有個名，同個目錄一致。有段描述，係驅動自動選取嗰個「幾時用」嘅觸發器，所以佢要具體。仲有一個來源欄位，第一方嘅 skills 會標做 ECC，而引入同貢獻嘅就會標做 community。個正文應該包含實用嘅指引、經測試嘅範例，同一個清晰嘅「幾時用」部分。一個 skill 資料夾可以裝多過淨係個主檔案，包括額外嘅參考檔案、範本，同 codemaps。

我哋剖析一個真正嘅 skill：測試驅動開發工作流程。佢係 ECC 入面最重要嘅其中一個，亦都係寫你自己嗰個嘅好範本。佢嘅結構係咁。第一，幾時啟動：新功能、修 bug、重構、新嘅端點或者組件。第二，核心原則：程式碼之前先寫測試、八成以上嘅覆蓋率，同三種測試類型，即係單元、整合同端對端。第三，git 檢查點：每個階段之後 commit，喺訊息裏面附上證據，喺當前嘅分支上面。第四，七個步驟：寫用戶旅程、產生測試案例、執行測試並確認佢哋失敗、實作最小嘅程式碼、執行測試並確認佢哋通過、重構，同驗證覆蓋率。第五，紅色關卡：一個關於乜嘢先算係一個有效嘅失敗測試嘅嚴謹定義，無論係運行時失敗定編譯時失敗，咁你就唔可以做假。正如個 skill 所講，一個淨係寫咗、但冇編譯同執行過嘅測試，唔算係紅色。第六，測試模式：畀單元測試框架、API 整合，同瀏覽器測試嘅具體範例。而第七，模擬範例、覆蓋率門檻、要避免嘅常見錯誤，同成功指標。留意吓個 skill 有幾多係關於嚴謹。佢唔係淨係話「寫測試」。佢精確咁定義咗乜嘢係一個有效嘅失敗測試，並且拒絕等你跳過紅色嗰步。嗰種精準，就係令一個 skill 可信到足以委派畀一個 agent 嘅原因。

而家我哋遊覽吓個 skill 目錄，唔係一份完整清單，而係感受吓啲類別。有超過 260 個 skills，所以背佢哋係冇意思，但知道啲群組就好強大。喺工程核心，你會搵到關於測試驅動開發、驗證回圈、評估框架、安全審查同掃描、編程標準、API 設計、後端同前端模式、資料庫遷移、部署同 Docker 模式、端對端測試、錯誤處理，同 git 工作流程嘅 skills。有語言同框架包，每個都為一個指定嘅技術棧捆綁咗模式、安全、測試驅動開發同驗證，涵蓋 Python、Go、Rust、Java、Kotlin、Swift、C++、Perl、Django、Laravel、NestJS、React、Next.js、Bun、Flutter 同 Android。有一個關於上下文同成本工程嘅群組，有關於策略性壓縮、上下文預算、token 預算建議、迭代式檢索、成本感知流水線、內容雜湊快取，同喺處理結構化文字時揀正規表達式定 AI 模型嘅 skills。有一個學習同元層級嘅群組，有持續學習、skill 盤點、rules 提煉、agent 評估、框架建構、程式碼庫上手，同設定 ECC 本身。有一個編排同回圈嘅群組，有自主回圈、持續代理回圈、多工工作流程，同計劃編排。而仲有一個研究、內容同操作者嘅群組，有深度研究、搜尋優先、市場研究、文章寫作、內容引擎、跨平台轉發、投資者材料、簡報製作，同影片剪輯，再加醫療、物流、海關、能源同製造嘅領域包，同一個預測市場包。呢個遊覽嘅重點係：無論你喺度做乜，好可能都有一個 skill 啱用。用 consult command 去搵佢。

快速講一講 skills 存喺邊，即係放置政策。ECC 區分精選嘅 skills，佢哋住喺 repo，同生成或者引入嘅 skills，佢哋住喺你嘅家目錄設定下面。政策係咁。Claude Code 淨係由你 skills 目錄嘅直接子項目載入 skills，所以手動安裝唔好巢狀放喺一個子資料夾入面。畀做手動安裝嘅新用戶：複製核心同通用嘅 skills，加埋搜尋優先嗰個 skill，淨係喺需要時先加冷門嘅包。而生成嘅 skills，即係透過 skill-create command 建立嘅，連同學到嘅 skills，會落喺你嘅代理資料家目錄下面。如果你想要完整嘅規則，repo 入面有關於放置同 skill 開發嘅權威文件。

你點樣建立 skills 呢？有兩條路。手動嗰條：建立一個帶主 skill 檔案嘅目錄，畀佢一個帶名、描述同來源嘅標頭，然後寫一個帶「幾時用」部分、步驟同經測試範例嘅正文，再用提供嘅腳本去驗證佢。又或者生成嗰條，即係 ECC 開採你嘅 repo。你執行 skill-create command 去分析當前嘅 repo 並產出 skill 檔案，而你可以加一個旗標，順便為持續學習產生 instincts。佢喺本機讀取你嘅 commit 歷史，並且抽取可重用嘅模式。仲有一條進階嘅 GitHub App 路線，畀超大型 repo 同團隊分享用。

咁，重點精華。Skills 係標準、耐用嘅工作流程介面，而新邏輯就屬於呢度。一個 skill 係一個帶主檔案嘅目錄，裝住一個標頭、一個正文，同可選嘅 codemaps。好嘅 skills 係嚴謹嘅；測試驅動開發工作流程甚至定義咗乜嘢先算一個有效嘅失敗測試。有超過 260 個 skills，橫跨工程、語言包、上下文工程、學習、編排，同操作者領域。而你用 consult command 去搵 skills，並且用手動方式或者 skill-create command 去建立佢哋。
