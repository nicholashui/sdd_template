Chapter fourteen. Token optimization and performance.

Agentic coding can get expensive fast, and quality actually degrades as you fill the context window. ECC treats cost and context as engineering problems with real levers. This chapter collects them.

Let's start with the settings that move the needle. In your Claude settings file, a few choices matter most. The model setting defaults to the most powerful model, but the recommendation is to use the balanced model. That cuts cost by roughly sixty percent and handles more than eighty percent of coding tasks. The max-thinking-tokens setting defaults to nearly thirty-two thousand, but dropping it to ten thousand cuts the hidden thinking cost per request by about seventy percent. The auto-compaction percentage defaults to ninety-five, but setting it to fifty compacts earlier, which gives better quality in long sessions. And subscription users can turn off the cost warnings, which hides the API-rate estimates while keeping the context, scope, and loop warnings. You switch up to the most powerful model only when you need depth.

The single biggest cost lever is model routing: matching the model to the task. In other words, not using the most powerful model for everything. Here's ECC's recommended routing. For exploration and search, use the cheapest model, because it's fast, cheap, and fine for finding files. For simple single-file edits, the cheapest model again, since instructions are clear and risk is low. For multi-file implementation, the balanced model, which is the best balance for coding. For pull-request reviews, the balanced model, which catches nuance affordably. For complex architecture, the most powerful model, for deep reasoning. For security analysis, the most powerful model, because you can't afford to miss things. For debugging hard bugs, the most powerful model, since you must hold the whole system in mind. And for writing docs, the cheapest model, because the structure is simple. The headline: default to the balanced model for about ninety percent of coding. Escalate to the most powerful model only when the first attempt failed, the task spans five or more files, it's an architectural decision, or it's security-critical. This is also why sub-agents matter so much: a sub-agent architecture lets you assign the cheapest model sufficient for each sub-task, instead of paying top rates for a file search. ECC's model-route command and harness-optimizer agent help automate this.

Next, protect the context window, because quality falls off a cliff near the limit. Guard it several ways. Avoid the last twenty percent or so of the window for large refactors and multi-file features, though single edits and docs tolerate higher utilization. Keep MCPs lean, fewer than ten servers and fewer than eighty tools enabled, which alone can reclaim more than a hundred thousand tokens. Cap the session-start context, or turn it off entirely for local, low-context models. Use strategic compaction: disable auto-compaction and compact at logical breaks. ECC has a hook that nudges you, and the pre-compact hook saves state first. Use the context-budget command to see where tokens are going and trim overhead. And lean on the context monitor, the hook that warns about exhaustion, runaway cost, scope creep, and tool loops.

There are also tool-level token savings. Use semantic search instead of plain text search. In ECC's fifty-task benchmark, semantic search with Claude Code used roughly half the tokens at similar or better judged quality. Keep codebases modular, with files in the hundreds of lines rather than thousands, which are cheaper to read and succeed on the first try more often, and this is also a rule. Use a command-line tool wrapped in a skill instead of an always-on MCP, which keeps tool descriptions out of context. And use codemaps, letting skills carry lightweight maps so the assistant navigates without re-exploring.

On parallelization, which the everyday-workflows chapter covered, the summary is this: parallel work multiplies throughput, but ECC advises the minimum viable amount. Use a fork for non-overlapping, read-only work like questions and research, while the main chat edits code. Use git worktrees for overlapping code work, with one instance per checkout, and rename your chats. And use the cascade method, focusing on three or four tasks at most, from oldest to newest. The guidance against running ten terminals just because you can: every extra instance is context you have to manage. Add one only out of necessity.

There's also verification efficiency. Re-doing work because you didn't verify is its own cost. ECC's eval framing: run checkpoint evals at logical milestones, and continuous evals after major changes. Use pass-at-k when you just need it to work, and pass-to-the-k when you need consistency. The evaluation-harness and verification-loop skills, plus the quality gate, implement this. Catching a regression early is far cheaper than catching it after it ships.

For cost visibility, there's a cost-report command for a cost summary, and the context-budget command for token overhead analysis. The dashboard and status tools surface metrics so cost isn't a mystery, and a metrics-bridge hook keeps a running session aggregate for the status line and context monitor.

If you want a single fast-and-cheap starter config to begin with, here it is in words. In your settings, use the balanced model, set max-thinking-tokens to ten thousand, and set auto-compaction to fifty percent. Then set your hook profile to standard, cap the session-start context, keep fewer than ten MCPs and fewer than eighty tools enabled per project, default to the balanced model and escalate to the most powerful only when truly needed, use semantic search, keep files modular, and compact at logical breaks.

So, the key takeaways. Set the balanced model, ten thousand thinking tokens, and fifty-percent auto-compaction for big cost cuts. Route models by task: cheapest for search, balanced as the default, most powerful only for hard or critical work. Protect context: fewer than ten MCPs and eighty tools, cap the session start, compact strategically, and watch the context budget. Semantic search and modular files cut token usage materially. And parallelize minimally, while verifying early, with checkpoint and continuous evals and pass-at-k versus pass-to-the-k thinking.
