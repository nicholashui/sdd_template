Chapter ten. MCP and context management.

Let's start with what MCP is. MCP, the Model Context Protocol, is a standard way to connect your assistant to external services: GitHub, a database, a deployment platform, a browser. An MCP server exposes tools the model can call directly. Instead of you copy-pasting a database result into chat, the model queries the database itself. The shortform guide frames it well: an MCP is a prompt-driven wrapper around an API. It's not a replacement for the API, but a flexible way for the model to navigate it. For example, a database MCP lets the assistant list tables and run queries without leaving the session.

ECC keeps its MCP configs in one file and ships exactly one default connector, for Chrome developer tools, which provides browser control. Everything else is opt-in, either a skill that wraps a command-line tool or REST API, or a catalog entry you enable deliberately. This is a documented policy, and it exists to protect the thing this chapter is really about: your context window.

Here is the most important operational lesson in all of ECC, and it surprises almost everyone, so read it twice. Every MCP tool description consumes tokens from your context window, before you've done any work. Too many enabled MCPs can shrink a two-hundred-thousand-token window down to around seventy thousand, and quality degrades hard as you approach the limit. Each enabled MCP server advertises its tools, and every tool's description sits in context permanently. Enable a dozen chatty MCP servers, and you've spent a third of your window on menus you may never use.

So here's the rule of thumb, straight from the guides. Keep fewer than ten MCP servers enabled. Keep fewer than eighty tools active. It's perfectly fine to have twenty or thirty MCPs configured. Just disable the ones you're not using on a per-project basis. You can see what's enabled with the MCP command, and disable the unused ones there. One note: the MCP command is the reliable live toggle. Editing your settings file is not a dependable way to turn off already-loaded MCP servers, so use the MCP command.

There's a sharp optimization the longform guide shares: the CLI-instead-of-MCP trick. Many MCPs are replaceable. Version control like GitHub, databases, and deployment platforms already have excellent command-line tools that the MCP is essentially wrapping. So instead of loading an MCP at all times, build a skill or command that wraps the command-line tool you need. For example, a pull-request command that runs the GitHub CLI with your preferred flags. You get the convenience without the always-on context cost. This is the recommended default for anything with a good command-line tool. Reserve MCP for services where the tool-call ergonomics genuinely help.

How do you configure MCP in ECC? Because plugin installs don't auto-enable bundled MCPs, you opt in. You can do it live in Claude Code with the MCP command, which adds, enables, or disables servers and persists the change. Or, for repo-local access, you copy the servers you want into a project config file, replacing any placeholder keys with real ones. And if you already run your own copies of ECC-bundled MCPs, you set an environment variable listing them, so the install and sync flows skip them rather than re-adding duplicates. That environment variable is an install and sync filter, not a live toggle. The set of MCP servers ECC knows about, where you'd enable just a handful per project, includes GitHub, a database connector, a live-docs service, a neural search service, memory, a browser-automation tool, a sequential-thinking helper, several deployment platforms, an analytics database, a web-crawling service, and the default Chrome developer tools connector.

Context discipline isn't only about MCP, though. ECC and the guides recommend several other habits. First, use a semantic search tool instead of plain text search. In ECC's fifty-task benchmark, the semantic search used roughly half the tokens at similar or better quality. Second, keep codebases modular, with files in the hundreds of lines rather than thousands. Smaller files mean cheaper reads and better first-try success, and this is also a rule. Third, use strategic compaction: disable auto-compaction and compact at logical intervals instead. ECC has a hook that nudges you at good moments, and the pre-compact hook saves state first. Fourth, lean on the context monitor, the hook that warns you about context exhaustion, runaway cost, scope creep, and tool loops before they bite. And fifth, use the context-budget command to analyze where your tokens are going and trim the overhead. The session-start context injection is itself capped, at a sensible default, and you can tune it or turn it off entirely for low-context setups.

There's a close cousin of MCP worth mentioning: plugins. Plugins package tools for easy install, and a single plugin can bundle skills, hooks, MCP servers, or language-server integrations. The same context warning applies. Enabling many plugins, along with their tools, eats your window. The author keeps around fourteen plugins installed but only four or five enabled at a time. Particularly useful plugin types include language-server plugins, which give real-time type-checking and go-to-definition when you run the assistant outside an IDE; the hookify plugin for creating hooks conversationally; the semantic-search plugin; and a live-documentation plugin.

So, the key takeaways. MCP connects the model to external services as callable tools, and ECC ships only the Chrome developer tools connector by default. Every enabled MCP costs context tokens up front, so keep fewer than ten servers and fewer than eighty tools enabled: configure many, enable few. Use the MCP command as the live toggle, not the settings file. Prefer a command-line tool wrapped in a skill over an always-on MCP for services with good command-line tools. And protect your context with semantic search, modular files, strategic compaction, the context monitor, and the context-budget command.
