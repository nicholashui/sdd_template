Chapter thirteen. Continuous learning.

Let's start with the problem this solves. You've all felt this: you correct the assistant about the same thing three days in a row. It hits the same bug, you give it the same workaround, it forgets, and you repeat. That's wasted tokens, wasted context, and wasted time. ECC's answer is this: when the assistant discovers something non-trivial, a debugging technique, a project-specific quirk, a workaround, it should save that as reusable knowledge, so next time the pattern loads automatically. The catalog grows from your own usage.

It's worth not conflating memory and learning, because they're different. Memory, from the earlier chapter, is this project's state carried between sessions. Continuous learning is about durable patterns extracted into instincts and skills that apply across sessions and across projects.

ECC ships two generations of this, and you should know which you're using. The first generation uses a stop hook to extract patterns into learned skills. It's simple and proven. The second generation is instinct-based, with confidence scoring, plus import, export, and evolution. Keep the first generation only if you explicitly want that legacy learned-skill flow. Otherwise, prefer the second generation. The second-generation system is the recommended path.

Let's talk about instincts, the unit of learning. An instinct is a small, scored piece of learned behavior: an observation plus an action, with a confidence score and supporting evidence and examples. Instincts accumulate quietly as you work, because observation hooks capture tool-use signals, and you manage them with commands. There's an instinct-status command to show your learned instincts with their confidence scores, both project-level and global. There's an instinct-import command to bring in instincts from a file, a URL, or a teammate. There's an instinct-export command to share yours. There's a prune command to delete expired pending instincts, which have a thirty-day time-to-live. There's a promote command to lift project-scoped instincts to global scope. And there's a projects command to list known projects and their instinct stats. Instincts are scoped: an instinct learned in one project stays project-scoped until you promote it to global. This keeps project quirks from leaking everywhere, while still letting genuinely universal lessons graduate. As a note on internals, instinct storage is separate from session memory, so clearing your sessions doesn't wipe what you've learned.

Individual instincts are useful, but the real magic is clustering related instincts into a brand-new skill, and that's what the evolve command does. The flow goes like this: a learn command extracts reusable patterns from the current session, a learn-and-evaluate command extracts patterns and self-evaluates their quality before saving, and the evolve command analyzes your learned instincts and suggests evolved skill structures. End to end, the loop is: a repeated pattern is observed by the hooks, it's saved as an instinct with a confidence score, the evolve command clusters related instincts into a new skill, and that new skill auto-loads next time the situation matches. This is how a generic assistant becomes your assistant: the things you keep teaching it congeal into skills that are always there.

There's a complementary path that doesn't wait for instincts to accumulate: generating skills from your git history. You run the skill-create command to analyze the current repo's commit history and produce skill files, and you can add a flag to also generate instincts for the second-generation learning system. It reads your commit history locally and extracts patterns into ready-to-use skills. For very large repos, team sharing, or automatic pull requests, there's an advanced GitHub App route, but the skill-create command covers the common case with no external service. You can also audit the health of your accumulated knowledge: there's a skill-health dashboard with analytics, a skill-stocktake to audit skills and commands for quality, and a rules-distill command that scans skills, extracts cross-cutting principles, and distills them into rules.

It's worth repeating why capture runs on the stop event, not on every prompt. It's a thoughtful design choice. Learning capture runs once at session end, rather than on every message, because running it per-message would add latency to every single prompt. Stop runs once, lightweight, and out of your way. The observation hooks do capture lightweight signals during the session, asynchronously and with a short timeout, but the heavy extraction happens at the end.

You don't need to do all of this on day one. Here's a sane adoption ramp. In week one, just work. Let instincts accumulate, and occasionally run instinct-status to see what it's noticing. In week two, at the end of meaty sessions, run the learn-and-evaluate command, then save the session. In week three, run the evolve command to turn clusters of instincts into skills, and run skill-health to check quality. And on an ongoing basis, promote the genuinely universal lessons to global, export instincts to share with teammates, and prune to keep things tidy.

So, the key takeaways. Continuous learning turns repeated corrections into durable, auto-loading knowledge. Prefer the second-generation system, with instincts and confidence scores, over the first-generation stop-hook flow. Manage instincts with the status, import, export, prune, and promote commands, and remember instincts are project-scoped until promoted. The evolve command clusters instincts into new skills, while the skill-create command mines your git history. And capture runs on the stop hook to avoid per-prompt latency.
