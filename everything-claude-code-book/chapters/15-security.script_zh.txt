第十五章。安全。

在你启用会执行 shell 命令的 hooks、连接 MCP 服务器，或者放手让一个 agent 自主循环之前，先看这一章。ECC 内置一份完整的安全指南是有原因的：代理式工具已经变成一个真实的攻击面。

我们讲清楚为什么这不是妄想。安全指南以一个令人清醒的论点开头：我们信任的工具，也是被攻击目标的工具。持续运行的框架增加了攻击面，而代理系统里面的提示注入，已经不再是一张好笑的越狱截图。它可以变成 shell 执行、机密信息外泄、工作流程滥用，或者悄无声息的横向移动。指南里面几个数字锚定了风险，而这些是一张快照。有一个 Claude Code 信任前执行的问题，在严重程度尺上评为八点七。有一份 Microsoft 关于 AI 记忆中毒的报告，横跨十四个行业的三十一家公司。有一份研究发现，在接近四千个公开 skills 之中，有三成六含有提示注入，识别出差不多一千五百个恶意载荷。还有一份报告讲超过一万七千个外露的代理框架实例。确切的数目会变。重要的是发展的方向。

最干净的心智模型，是一个叫做致命三重奏的东西，由 Simon Willison 提出并且在指南引用。有三种成分，只在同一个运行时里面结合才危险。第一，私人数据：你的机密信息、repos 和凭证。第二，不可信内容：一个拉取请求的留言、一个邮件附件、一个抓取的网页、一个 MCP 工具的输出。而第三，对外通讯：把数据送出去的能力。任何一样或者两样，通常都没问题。三样加在一起，就是提示注入变成数据外泄的时候。模型读的每一个字节，都是可执行的上下文。一旦文本进入窗口，数据和指令之间就没了一条清晰的界线。

你实际上面对什么攻击向量？嵌入了指令的邮件和 PDF 附件，而你的 agent 把它们当成任务的一部分来读。带隐藏差异留言或者恶意链接文档的 GitHub 拉取请求和议题，这样危险是因为自动审查机器人会把漏洞向下游传播。有漏洞、恶意，或者纯粹被过度信任的 MCP 服务器；OWASP 现在有一个 MCP 十大，涵盖像工具中毒、上下文载荷注入、命令注入、影子服务器，和机密信息外泄的东西。记忆中毒，在那里一个载荷被记住，然后稍后再浮现。以及把 skills 当成一条供应链，因为一份研究里面有三分之一的公开 skills 带有注入，所以把引入的 skills 当成任何依赖一样对待。

有一些真实世界的漏洞，结束了“这样东西夸大其词”的阶段。Check Point Research 披露了几个 Claude Code 的问题。其中一个，项目内含的代码，可以在信任对话框被接受之前运行。另一个，一个恶意项目可以覆写 API base-URL 变量、重定向 API 流量，并且在信任确认之前泄露 API 密钥。而第三个，repo 控制的 MCP 设置，可以在那个目录被有意义地信任之前，自动批准项目的 MCP 服务器。ECC 得出的教训是这样：项目配置、hooks、MCP 设置，和环境变量，全部都是执行面的一部分。你的配置文件通过版本控制分享，并且由一条信任边界把守，而这正正是攻击者的目标。这也是为什么你应该对 clone 和打开不可信 repos 保持谨慎。


指南对纵深防御讲得令人耳目一新地具体。有五层，最容易见到投资回报的排第一。第一，分隔身份。不要把你的账号给 agent。创建一个独立的机器人用户，配一个短期、有范围的 token。如果你的 agent 有和你一样的账号，一个被入侵的 agent 就是你。第二，隔离不可信的工作。在一个容器、一个虚拟机，或者一个 devcontainer 里面运行不可信的 repos、附件多的任务，或者外来内容的工作，并且默认没有网络出口，这样一个被入侵的 agent 就没法打电话回家。即使是一个一次性审查用的、只是没有网络的容器，都好过在你的主机上运行。第三，限制工具和路径，这是最高回报也是最容易的。如果你的框架支持权限，从围住明显敏感物料的拒绝 rules 开始：你的 SSH 密钥、你的云端凭证、你的环境文件，和危险的 shell 命令，例如把一个下载直接导入一个 shell，或者用远程访问工具。只给一个工作流程它需要的东西。一个读一个 repo 然后测试的工作，不应该读你的主目录，而一个单一 repo 的 token，不应该有全组织的写入权限。第四，净化输入。模型读的一切都是可执行的上下文，所以留意隐藏的 unicode、零宽度或者双向字符、藏在 HTML 注释里面的载荷，和嵌入在附件或者链接内容里面的指令。在模型看到附件之前先净化它们。ECC 的提示防御基线，就是这样东西在提示里面的一层。而第五，设置批准边界、记录，和紧急停止开关。为改变状态的动作保持有人在循环里面、记录工具和 MCP 调用，并且有一个紧急停止开关。默认是最少代理权。

ECC 内置一个叫做 AgentShield 的扫描器，是一个给你的代理配置用的安全审计器，零安装就可以运行。有一个快速扫描、一个给安全问题用的自动修复选项、一个深度三代理分析模式，和一个从零开始生成一个安全配置的 init 选项。它扫描什么？你的指令文件、你的设置、MCP 配置、hooks、agent 定义，和 skills，横跨五个类别：用十四个模式的机密信息检测、权限审计、hook 注入分析、MCP 服务器风险剖析，和 agent 配置审查。深度模式运行三个强大的 agents，做一条红队、蓝队和审计员的流水线：攻击者找漏洞链、防守者评估保护措施，而审计员综合一个有优先次序的风险评估。那是对抗性推理，不是只是模式匹配。输出以一个从 A 到 F 的终端评级来给，再加机器可读、Markdown 和 HTML 格式，而它在关键发现上交回一个失败退出码，这样你就可以把守构建。在框架里面，你把它当成一个安全扫描来运行，而在持续集成里面，你用提供的 action。AgentShield 在一个 Claude Code 黑客松做出来，并且报告超过一千二百个测试、大约九成八的覆盖率，和超过一百条静态分析规则。

这里是最低门槛的清单，在你放手让 ECC、或者任何代理框架，用真实权限运行之前。这个 agent 用一个独立的身份，不是你的个人账号。不可信的工作在一个没有默认出口的隔离容器或者虚拟机里面运行。拒绝 rules 保护你的 SSH 密钥、云端凭证、环境文件，和危险的 shell 命令。工具限定在每一个工作流程需要的东西，而 tokens 是短期和最小权限。输入经过净化，而提示防御基线存在。改变状态的动作需要批准、工具和 MCP 调用有记录，而你有一个紧急停止开关。你已经运行了 AgentShield 扫描并且修了那些关键问题。你启用了少于十个 MCP 和少于八十个工具，而你信任每一个。而你不在你的主机上 clone 和打开不可信的 repos。

ECC 不只是关于安全的建议。它烘焙了很多东西进去。提示防御基线在每一个 agent 和主指令文件里面。安全优先是一条核心原则，有强制的 commit 前检查：没有硬编码的机密信息、输入验证、防止常见的网络漏洞、验证检查、速率限制，和日志里面没有敏感数据。有一个专门的 security-reviewer agent，和一个跟 OWASP 对齐的 security-review skill。有给机密信息检测、配置保护、治理捕捉，和 MCP 健康用的 hooks。还有一个供应链入侵指标扫描，和咨询来源的工具。

那么，重点精华。致命三重奏，私人数据加不可信内容加对外通讯，就是注入变成外泄的时候。项目配置、hooks、MCP 设置，和环境变量，都是执行面，而真实世界的漏洞证明了这一点。纵深防御：分隔身份、隔离、限制工具和路径、净化，和批准、记录和停止。运行 AgentShield 扫描、用深度模式做对抗性分析，并且在失败退出码上把守持续集成。并且把 skills 和 MCP 服务器当成供应链，不要在你的主机上 clone 和打开不可信的 repos。
