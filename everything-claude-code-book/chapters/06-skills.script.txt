Chapter six. Skills.

Let's talk about why skills are the heart of ECC. If agents are who does the work, skills are how the work is done. A skill is a reusable, self-contained workflow: the steps, the structure, the examples, the gotchas, and sometimes supporting files and codemaps. A codemap is a lightweight map of a codebase, so the assistant can navigate without burning tokens re-exploring.

ECC is emphatic that skills are the canonical workflow surface. New workflow development should land in skills first. Commands still exist for compatibility, but the durable logic belongs in skills. The reason traces back to the compounding-workflows philosophy from earlier: a skill is reused forever, across projects and sessions, and it gets better as models improve.

There are three ways a skill gets used. First, you invoke it by name. Second, it gets auto-suggested when its description matches what you're doing. And third, an agent reuses it: a sub-agent that can execute a subset of your skills can be handed a task and use them autonomously.

Now let's look at the anatomy of a skill file. Each skill lives in its own directory, with a main skill file inside. Its header is small. There's a name, which matches the directory. There's a description, which is the when-to-use trigger that drives auto-selection, so it needs to be concrete. And there's an origin field, which is marked as ECC for first-party skills, or community for imported and contributed ones. The body should include practical guidance, tested examples, and a clear when-to-use section. A skill folder can hold more than just the main file, including additional reference files, templates, and codemaps.

Let's dissect a real skill: the test-driven-development workflow. It's one of the most important in ECC and a great template for your own. Its structure goes like this. First, when to activate: new features, bug fixes, refactors, new endpoints or components. Second, core principles: tests before code, eighty-percent-plus coverage, and the three test types, unit, integration, and end-to-end. Third, git checkpoints: commit after each stage, with evidence in the message, on the current branch. Fourth, the seven steps: write user journeys, generate test cases, run the tests and confirm they fail, implement the minimal code, run the tests and confirm they pass, refactor, and verify coverage. Fifth, the red gate: a careful definition of what counts as a valid failing test, whether that's a runtime failure or a compile-time failure, so you can't fake it. As the skill puts it, a test that was only written but not compiled and executed does not count as red. Sixth, testing patterns: concrete examples for unit-test frameworks, API integration, and browser testing. And seventh, mocking examples, coverage thresholds, common mistakes to avoid, and success metrics. Notice how much of the skill is about rigor. It doesn't just say write tests. It defines exactly what a valid failing test is, and refuses to let you skip the red step. That precision is what makes a skill trustworthy enough to delegate to an agent.

Now let's take a tour of the skill catalog, not a full list, but a sense of the categories. There are more than two hundred and sixty skills, so memorizing them is pointless, but knowing the clusters is powerful. In the engineering core, you'll find skills for test-driven development, verification loops, evaluation harnesses, security review and scanning, coding standards, API design, backend and frontend patterns, database migrations, deployment and Docker patterns, end-to-end testing, error handling, and git workflow. There are language and framework packs, each bundling patterns, security, test-driven development, and verification for a given stack, covering Python, Go, Rust, Java, Kotlin, Swift, C-plus-plus, Perl, Django, Laravel, NestJS, React, Next.js, Bun, Flutter, and Android. There's a cluster for context and cost engineering, with skills for strategic compaction, context budgeting, token-budget advice, iterative retrieval, cost-aware pipelines, content-hash caching, and choosing regex versus an AI model for structured text. There's a learning-and-meta cluster, with continuous learning, skill stocktaking, rules distillation, agent evaluation, harness construction, codebase onboarding, and configuring ECC itself. There's an orchestration-and-loops cluster, with autonomous loops, continuous agent loops, multiplexed workflows, and plan orchestration. And there's a research, content, and operator cluster, with deep research, search-first, market research, article writing, content engines, crossposting, investor materials, slide building, and video editing, plus domain packs for healthcare, logistics, customs, energy, and manufacturing, and a prediction-market pack. The point of the tour is this: whatever you're doing, there's likely a skill for it. Use the consult command to find it.

A quick word on where skills are stored, the placement policy. ECC distinguishes curated skills, which live in the repository, from generated or imported skills, which live under your home config. The policy is this. Claude Code loads skills only from direct children of your skills directory, so don't nest manual installs in a subfolder. For new users doing a manual install, copy the core and general skills plus the search-first skill, and add niche packs only when needed. And generated skills, created through the skill-create command, along with learned skills, land under your agent data home. The repository has authoritative docs on placement and skill development if you want the full rules.

How do you create skills? There are two routes. The manual route: create a directory with a main skill file, give it a header with a name, description, and origin, and write a body with a when-to-use section, steps, and tested examples, then validate it with the provided script. Or the generated route, where ECC mines your repository. You run the skill-create command to analyze the current repo and produce skill files, and you can add a flag to also generate instincts for continuous learning. This reads your commit history locally and extracts reusable patterns. There's also an advanced GitHub App route for very large repos and team sharing.

So, the key takeaways. Skills are the canonical, durable workflow surface, and new logic belongs here. A skill is a directory with a main file, holding a header, a body, and optional codemaps. Great skills are rigorous; the test-driven-development workflow even defines what a valid failing test is. There are more than two hundred and sixty skills across engineering, language packs, context engineering, learning, orchestration, and operator domains. And you find skills with the consult command, and create them manually or with the skill-create command.
