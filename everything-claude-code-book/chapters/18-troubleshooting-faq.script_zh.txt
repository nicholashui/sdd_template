第十八章。疑难排解和常见问题。

这是一份关于你实际上会撞到的问题的实战指南，大致按它们有多常见排序。Repo 自己的疑难排解文档和常见问题，启发了这一章。

第一个问题：所有东西看起来都重复了，或者 ECC 感觉烦扰或者坏了。原因是你叠用了安装方法，跑了插件和完整安装器。插件已经加载了 skills、commands 和 hooks，而安装器又再复制了它们一次。这里是修复方法，按顺序。列出装了什么。执行 doctor command 去诊断。执行 repair command 去还原受管理的文件。而如果还是乱，就执行卸载，先用试运行模式，然后才真正执行。之后，移除插件安装、通过 repo 卸载、删除额外手动复制的 rule 文件夹，然后重新安装一次，只用一条路线。

第二个问题：一个 hooks 文件重复的错误，或者 hooks 不运作。原因是 Claude Code 2.1 版和之后，会自动加载一个插件的 hooks 配置。你要么在插件清单里面声明了一个 hooks 字段，要么在一个插件安装之上把 hooks 配置复制进了你的配置文件。修复方法：不要在插件清单里面加一个 hooks 字段，一个回归测试会强制执行这一点；插件用户不应该把 hooks 复制进他们的配置文件；而手动用户应该通过安装器、用 hooks-runtime 模块来安装 hooks。

第三个问题：你的上下文窗口在缩，或者助手用完了上下文。原因是太多启用了的 MCP 服务器和工具，因为每一个工具描述都吃你的窗口。一个二十万 token 的窗口，可以跌到大约七万。修复方法：用 MCP command 去停用没用的服务器。保持少于十个 MCP 和少于八十个工具。还有封顶会话开始的上下文，或者关闭它、策略性地压缩，并且检查 context budget。而记住，编辑你的配置文件，不是一个可靠的方法去停用已经加载的 MCP，所以用 MCP command。

第四个问题：hooks 太吵、太严格，或者太慢。在运行时调校它们，不用编辑文件。把 hook 配置设成 minimal 求安静些，或者 strict 求最大。用它们的 id 停用特定的 hooks。或者保留那些警告但丢掉成本估算。或者，安装 minimal 配置文件，它没有 hooks 运行时，然后稍后用 hooks-runtime 模块加回 hooks。

第五个问题：你装了插件，但你没有 rules。这是预期之内。Claude Code 插件无法分发 rules。手动复制它们：clone 那个 repo、在你的 Claude 配置下面建一个 rules 目录，并且复制进 common 的 rules 加你技术栈的语言包。复制整个语言目录，而不是单个的文件。

第六个问题：那些 multi commands 不运作。原因是它们需要一个基本安装不提供的外部工作流程运行时。修复方法是通过那个运行时的 npx command 去执行它的设置，这样就会设置好那些 commands 预期的包装和提示文件。

第七个问题：skills 或者 commands 没出现。检查安装，方法是列出插件的内容，或者列出手动装了什么。检查 skill 放置，因为 Claude Code 只从你 skills 目录的直接子项目加载 skills，所以不要把它们嵌套放在一个子文件夹。检查你的版本，确保命令行工具至少是 2.1.0 版。并且注意 Codex 插件路线在上游很脆弱，所以偏向用同步脚本，多过用 Codex 插件 marketplace。


第八个问题：你本机的 Claude 配置被清掉了，而你在想需不需要再买 ECC。不用。在重新安装之前，执行 list-installed command，接着 doctor 和 repair，这样通常就会还原 ECC 管理的文件。托管工具的账单和账号恢复，是和开源文件分开的另一回事。

第九个问题：Cursor 和 Claude Code 在互相覆盖对方的记忆。用一个环境变量，为 Cursor 设置一个独立的代理数据主目录。Cursor 的安装会尝试通过一个会话开始 hook 和一个覆写文件自动做这样东西。而如果你刻意想共用记忆，就把两个都指向你的 Claude 目录。

现在快速来一轮常见问题。可不可以只用部分组件，例如只用 agents？可以。做一个手动安装，并且复制你想要的，因为每一个组件都是独立的。它和你的 IDE 或者命令行工具运作得了吗？它原生支持 Claude Code，再加 Cursor、应用和命令行形式的 Codex、OpenCode、Gemini、Zed、GitHub Copilot、Antigravity、JoyCode 和 CodeBuddy、Qwen，和给其他工具的一条手动路线。可不可以在一个自定义 API 端点上跑 ECC？可以，设置 base-URL 和 auth-token 环境变量，因为一旦助手跑得起来，ECC 就和供应商无关，但要留意 base-URL 劫持的漏洞，所以只信任你自己的网关。你怎样检查装了什么？列出插件的内容，或者执行 list-installed command。你怎样贡献一个 skill 或者 agent？Fork 那个 repo、用对的头部加你的 skill 或者 agent 文件、运行测试，然后开一个拉取请求。它真的免费吗？这个 repo 永远是 MIT 许可证；托管的工具和赞助资助开发，但不是必需的。

当有些东西怪怪的时候，有一个好用的“第一样要执行”的顺序：列出装了什么、执行 doctor 去看哪里出错、执行 repair 去修受管理的文件、运行测试套件去检查这个 repo 本身健不健康，并且用 status command 拍一张完整的就绪度快照。

那么，重点精华。大部分问题都追溯到叠用安装或者双重加载 hooks，所以用 doctor 和 repair 去诊断，永远不要在上面再装。上下文问题就意味着削减 MCP 和封顶会话开始的上下文。用插件缺 rules 是预期之内，所以手动复制它们。Multi commands 需要它们独立的运行时。而当你拿不定主意：list-installed，然后 doctor，然后 repair。

最后，是参考那一章，把每一个术语、command 和环境变量放在一处。
