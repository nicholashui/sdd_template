第十一章。日常工作流程。

这一章你会不断回来看。它把所有组件变成具体、可重复的循环。复制它们、改编它们，它们就会变成肌肉记忆。

我们从主循环开始：规划，然后测试驱动开发，然后审查，然后验证，然后学习。差不多每一次 ECC 会话都遵循同样的形状。第一步是规划。你用像“加入 OAuth 登录”的东西去执行 plan command。Planner 代理产出一个计划，而你确认它。第二步是测试驱动开发。你调用测试驱动开发工作流程 skill，走红、绿、重构，由测试驱动指导带领。第三步是审查。你执行 code-review 和 security-scan commands，它们会调用 code reviewer 和 security reviewer。第四步是验证。你执行质量关卡、修任何构建错误，和确认测试覆盖率至少八成。而第五步是学习。你执行 learn-and-evaluate command 并且保存会话，产出一个 instinct 和一份会话摘要，还有停止 hook 帮忙。围着整件事，hooks 无形地格式化、类型检查和扫描机密信息，而 rules 约束每一步。在各阶段之间，清除上下文并且把中途的输出存成文件，这样你就不会淹没你的上下文窗口。

这里是开始一个新功能的食谱。你用像“用 OAuth 加入用户验证”的描述去执行 plan command。Planner 产出一个分阶段的实现蓝图，而你批准它。然后你调用测试驱动开发工作流程：测试驱动指导先写失败的测试，然后写最小的实现。然后你执行 code-review，而 code reviewer 检查那个差异，如果干净就交回零个发现。为什么是这个顺序？先规划后执行，阻止范围乱扩散。测试优先，意味着实现是由一个规格带领的。而立刻审查，就趁上下文还新鲜时捉到问题。

这里是修一个 bug 的食谱。你调用测试驱动开发工作流程，而测试驱动指导写一个重现那个 bug 的失败测试，一个有效的红色状态。然后你实现那个修复并且确认那个测试转绿。然后你执行 code-review 去捉回归。这里的纪律是之前那个红色关卡：在你动生产代码之前，你必须有一个真的因为预期的原因而失败的测试。没有重现，就不修。

这里是为生产做准备的食谱。你执行一个安全扫描，在那里 security reviewer 做一个跟 OWASP 标准对齐的审计，或者用 AgentShield。你调用端到端测试 skill，在那里端到端执行员在一个真实的浏览器里面演练关键的用户流程。你执行 test-coverage command 去验证八成以上的覆盖率。而你执行质量关卡，做一个对照项目标准的最后检查。


这里是用 sub-agents 编排一个更大型构建的食谱。对于多阶段的工作，依靠按顺序阶段的编排。阶段一是研究，产出一份研究摘要，用一个便宜的模型。阶段二是规划，产出一个计划，用一个强大的模型去做困难的设计。阶段三是实现，产出代码和测试，在一个平衡的模型上。阶段四是审查，产出一份审查。而阶段五是验证，它要么完成，要么循环回去失败那个阶段。编排 commands 把这些模式打包，让你构建一个 MVP、加入一个功能、修一个缺陷等等。而对于多模型协作，有那些 multi commands，不过记住它们需要装了它们独立的工作流程运行时。

我们讲一下并行化：一次性做多些。当一个会话不够用，ECC 的指引是刻意的，不是只开十个终端。第一，为不重叠的工作分叉。用一个 fork command 去为问题或者研究分叉一段对话，而你的主对话继续编辑代码。把代码改动留在主对话，用分叉去做只读的探索。第二，为重叠的代码工作用 git worktrees。当多个实例必须编辑重叠的代码，就给每一个它自己的检出，并且每个 worktree 跑一个实例，为你的对话改名，这样你就不会跟不上。第三，有级联方法：在右边的标签开新任务，从左扫到右，从最旧扫到最新，而且一次最多聚焦在三四个任务。指导原则是这条问题：用最低可行的并行量，你可以做到多少？只在真正必要的时候才加一个终端。

这里有一个给全新项目的好模式：双实例项目启动，在一个空 repo 上用两个助手实例。第一个实例是一个搭建代理，铺下项目结构和配置。第二个实例是一个深度研究代理，连接到服务和网络搜索、写产品需求文档、画架构图，和收集真实的文档。然后它们汇合：那个骨架遇上那个经过研究的计划。

接着是日终交接。不要让一个会话蒸发。捕捉它。执行 learn-and-evaluate command 去抽取可重用的模式，并且在保存之前自我评估它们，然后保存会话去为状态拍快照。或者依靠停止 hook 去自动写一份摘要。第二天早上，恢复会话去重新加载最新那个并且继续。而对于一个可携带、可分享的状态，很适合站会或者交接给队友，有一个 status command 写一个 markdown 交接文件。

ECC 把验证当成一个一等的工作流程，而不是事后诸葛。有两种评估风格。基于检查点的，在那里你设置明确的检查点、对照准则验证，并且在继续之前修复。和持续的，在那里你每隔几分钟或者在重大改动之后运行，用完整的测试套件和 linter。还有两种解读可靠性的方式。第一种是 pass-at-k，在那里 k 次尝试之中至少一次成功，你在只需要它跑得起来的时候用它。第二种是 pass-to-the-k，在那里 k 次尝试全部成功，你在一致性是必要的时候用它。评估框架和验证循环 skills 实现了这些，而质量关卡就是日常的关卡。

从指南来的几个高杠杆习惯，小小的东西但回报大。停用自动压缩，并且在合乎逻辑的断点策略性地压缩。在一个终端多路复用器里面运行长 commands，这样你就可以分离、重新连接，和流式查看日志。ECC 甚至会提醒你。用语义搜索代替纯文本搜索，把搜索 token 减半。设置一条状态栏，显示你的分支、上下文百分比、模型，和待办事项。逐个项目保持 MCP 精简。并且在 commit 之前审查差异，让 git-push hook 把守你。

那么，重点精华。主循环是规划、测试驱动开发、审查、验证、学习，全部包在 hooks 和 rules 里面。在修复之前，用一个有效的失败测试重现 bug。把大型工作编排成按顺序的阶段，在各 agents 之间传递文件。用分叉、git worktrees 和级联方法去并行化，但要最低限度。捕捉每一个会话，这样明天就赢在起跑线。而且用检查点和持续评估，加 pass-at-k 对 pass-to-the-k 的思维去验证。
