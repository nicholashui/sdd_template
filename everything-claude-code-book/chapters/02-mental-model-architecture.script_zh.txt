第二章。心智模型和架构。

要理解 ECC，而不是只死记，最快的方法就是把它看成四个叠起来的层。每一层回答一个不同的问题，而每一层都是由它上面那一层塑造出来的。

第一层是身份和治理。这就是 ECC 是谁。三个小小的根目录文件订立了宪法。有一个灵魂文件，装着核心身份和五大原则。有一个 rules 文件，装着硬性的“永远要”和“永远不要”约束，再加上 agents、skills 和 hooks 的必要格式。还有助手会读的指令文件，包括代理名册、安全指引和工作流程。这一层就是为什么其他所有东西会这样运作的原因。

第二层是组件目录。这是可重用的东西，是 repo 的核心，是你会安装和调用的东西。它分成 agents、skills、commands、rules，和 MCP 配置。本书大部分内容都是讲这一层。

第三层是框架适配器和安装器。这就是它怎样落到磁盘上。目录是通用的，但每一款 AI 工具都想它的配置放在不同的位置，用不同的格式，所以这一层负责翻译。有针对不同平台的安装入口。有一个由清单和安装脚本驱动的选择性安装引擎。一个状态存储记录了装了什么，这让干净的更新和卸载变得可能。而目标就是各工具的配置目录，给 Claude、Cursor、Codex、OpenCode、Gemini、Zed、GitHub 等等用。

第四层是运行时和强制执行。这就是你工作的时候正在发生的事。当一个会话真正运行的时候，这一层就是活的。一个 hooks 配置把事件对应到小小的脚本。那些 hooks 会格式化代码、扫描机密信息、把守质量、留存记忆，以及捕捉学习信号。而度量和可观测性，就喂给操作者命令行界面和 ECC 2.0 的控制平面。

这里有一个记住以上一切的心智捷径。身份治理目录。目录由适配器塑造。适配器把它落到磁盘。而运行时负责强制执行。上层治理下层。

现在我们游览一下 repo 的地图。你不会直接碰到大部分的东西，但了解一下地形，可以消除很多神秘感。在根目录，你有身份和治理的文件，再加一个版本文件。然后就是目录里面的各个文件夹：agents，有 64 个专门的子代理；skills，有超过 260 个，每个都在自己的文件夹；commands，有 84 个斜杠 commands，保持向后兼容但是技能优先；一个可选择加入的、已退役短 command 名的归档区；rules，有通用 rules 加各语言包；hooks；MCP 配置；和 contexts，会在开发、审查和研究模式里，把内容注入系统提示。

还有一个 scripts 文件夹，装着跨平台的管道工程：真正的 hook 实现、共用工具、操作者命令行界面，和选择性安装的规划器和应用器。有一个给 ECC 2.0 用的 Rust 控制平面原型。有一个桌面图形界面。有安装器入口、安装清单、用来验证的 JSON 结构、一套测试，以及像 SaaS、Go、Django 和 Rust 项目这样的示例配置。

Repo 里面有三份指南是纯金。Repo 讲得很白：这个 repo 只是源代码，而那些指南会解释一切。短篇指南涵盖基础、skills、hooks、subagents、MCP 和插件，你应该先看它。长篇指南涵盖 token 经济学、记忆留存、评估，和并行化。而安全指南涵盖攻击向量、沙箱、真实世界的漏洞，和 AgentShield。本书把这三份提炼和整理，但原作本身就很出色。

我们追踪一下一个单一请求怎样流过 ECC。假设你执行 plan command，请求加入 OAuth 登录。这里就是实际发生的事。第一，治理加载。Rules 和指令文件已经在上下文里，所以助手知道那些标准：测试驱动开发、不可变性、安全。第二，如果有装，一个会话开始 hook 已经注入了之前会话的记忆，并且检测了你的包管理器。第三，command 或者 skill 启动。Plan command 对应到规划逻辑，而这个逻辑可能会委派给 planner 代理。第四，代理在一个有明确范围的上下文里工作，用有限的工具和合适的模型，然后交回一个计划。第五，你确认，因为先规划后执行就意味着它会等你。第六，实现开始进行，很可能通过测试驱动开发 skill 和相关的指导代理。第七，hooks 不断启动：每次编辑都格式化、类型检查、机密信息扫描，以及对多余调试日志的警告。第八，审查通过 code-review command 和安全扫描进行。而第九，一个停止 hook 写下会话摘要，并且抽取任何学到的东西。每一层都有参与。那种编排就是整件事的重点。

有一个经常让人混淆的地方值得搞清楚：ECC 住在两个家。有源代码 repo，是一个 git clone，还有你安装好的配置，是住在你各工具的配置目录里。源代码 repo 包含完整的目录、脚本和指南。你安装好的配置只包含你实际装了的组件。你贡献或者定制的时候，你改的是源代码，而你调整你实时配置的时候，你改的是安装好的配置。源代码是通过拉取新改动来更新，而你安装好的配置是通过重新执行安装器或者更新插件来更新。安装器从源代码读取，把解析好的文件写入你的配置目录，而状态存储记住那个对应关系，这样 doctor、repair 和卸载工具就可以精准地运作。给插件用户的一点提示：用 Claude Code 插件的时候，源代码是一个受管理的缓存，而不是一个你要自己维护的 clone。如果你想复制那些 rules，你还是要另外 clone 那个 repo，因为插件无法分发 rules。

还有一些命名你会碰到，可能会让人混淆。ECC 有三个刻意做成不同的公开标识符。GitHub 源代码 repo 用一个名。Claude marketplace 插件用一个简短的标识符。而 npm 包就用另一个名。所以安装 commands 是刻意用不同的名。插件 id 保持简短，是为了满足严格的验证器，而 npm 包就保留了它原本的名。比较旧的帖子可能会显示一个比较长的旧版 marketplace id，你应该当它已经弃用。

那么，重点精华。ECC 是四层：身份、目录、适配器和运行时，上层治理下层。目录，由 agents、skills、commands、rules 和 MCP 配置组成，是你会安装和调用的东西。安装器、清单和状态存储，把那个通用目录翻译到每一款框架。运行时，由 hooks 和脚本组成，是实时强制执行标准的那个。有两个家：源代码 repo 和你安装好的配置。而三个不同的名，全部指的都是同一个项目。

现在你有了地图，我们就来第一次就正确地安装它。
