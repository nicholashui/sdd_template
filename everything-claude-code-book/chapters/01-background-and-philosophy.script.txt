Chapter one. Background and philosophy.

Let's start with where ECC came from. ECC is the work of Affaan Mustafa, who has been using Claude Code since its experimental rollout. The project grew out of real product engineering, most notably building zenith dot chat, which won the Anthropic and Forum Ventures hackathon. The repository's own framing is blunt about its origin. It says it's not just configs, but a complete system, evolved over more than ten months of intensive daily use building real products.

That detail matters. ECC is not a theoretical best-practices template someone wrote in a weekend. It's the fossil record of thousands of hours of agentic coding. Every hook, rule, and skill exists because some real failure mode made it necessary. When you read a rule like never commit a debug log, or a hook that blocks dev servers from running outside a terminal multiplexer, you are reading a scar.

The name is a small joke that became a big project: Everything Claude Code. It started Claude-centric, then expanded to cover the whole ecosystem of AI coding tools.

Here's a quick timeline drawn from the changelog. You don't need to memorize this, but it shows the direction of travel. Version one-point-two unified commands and skills, added session management, and introduced continuous learning with instincts. Version one-point-three brought full OpenCode plugin support. Version one-point-four added multi-language rules, an installation wizard, and multi-agent commands. Version one-point-six added Codex command-line support, the AgentShield security auditor, and a GitHub Marketplace app. Version one-point-seven expanded across platforms and added a presentation builder skill. Version one-point-eight reframed the whole thing as a harness performance system, with a hook reliability overhaul and runtime hook controls. Version one-point-nine introduced selective install, more language support, and a SQLite state store. And version two-point-oh became the agent harness operating system, with a control-plane substrate, a worktree lifecycle, an orchestrator family, and a Discord community. The throughline is clear: ECC started as a config pack and became an operating system for running agents.

So what exactly is an agent harness, and why optimize it? A harness is the program that wraps a language model and lets it actually do things: read files, run shell commands, edit code, call tools. Claude Code, Cursor, Codex, and OpenCode are all harnesses. The model is the engine. The harness is the car around it.

Here's the key insight ECC is built on. Most of the quality, cost, and safety of agentic coding is determined by the harness, not the model. Two people using the exact same model get wildly different results depending on a few things. What context the model sees, and how much of its window is wasted. Which sub-tasks get delegated to cheaper or specialized configurations. What gets checked automatically, like tests, formatting, and secrets, versus what's left to chance. What the model remembers between sessions. And what the model is allowed to do, in terms of tool permissions and sandboxing.

ECC is a harness performance system: a curated, battle-tested set of configurations that tune all of those levers at once. Here's a helpful analogy. The model is a brilliant new hire. The harness is their laptop and accounts. ECC is the onboarding handbook, the team of mentors, the testing pipeline, and the security policy. It's everything that turns raw talent into reliable output.

Now let's talk about the five core principles. Three small files at the root of the repository encode ECC's worldview: one for identity, one for hard constraints, and the instructions the assistant reads. Five principles recur everywhere.

Principle one: Agent-First. Route work to the right specialist as early as possible. Instead of one generalist doing everything, ECC pushes work to scoped sub-agents. A planner for strategy, a code reviewer for quality, a security reviewer for sensitive code, a build-error resolver when the toolchain breaks. This keeps each context small and each agent expert.

Principle two: Test-Driven. Write or refresh tests before trusting implementation changes. ECC mandates the red, green, refactor loop, and an eighty-percent-plus coverage target across unit, integration, and end-to-end tests. This is enforced both by a test-driven-development skill and by hooks that nudge you toward coverage.

Principle three: Security-First. Validate inputs, protect secrets, and keep safe defaults. Every agent carries a prompt-defense baseline. Hooks scan for secrets. There's a whole security guide and a dedicated scanner, and we'll devote an entire chapter to it.

Principle four: Immutability. Prefer explicit state transitions over mutation. This is a coding-style principle: create new objects rather than mutating existing ones. It shows up in the rules, in the reviewer agent's checklist, and in the example code throughout.

Principle five: Plan Before Execute. Complex changes should be broken into deliberate phases. The plan command and the planner agent exist to stop the assistant from charging into a twelve-file change with no blueprint. Plan, get confirmation, and then touch code.

These five are the constitution. If you ever wonder why ECC does something a certain way, it almost always traces back to one of these.

There's also a philosophy of compounding workflows. The longform guide quotes a line that captures ECC's deepest bet. Early on, the author spent time building reusable workflows and patterns. It was tedious to build, but it had a wild compounding effect as models and agent harnesses improved. The argument is this: the prompt you type today is thrown away. But the skill you write today keeps paying off, every session, on every project, and it gets better as the underlying models improve. So ECC invests heavily in durable artifacts: reusable specialists, reusable workflows, planning patterns, and context-engineering patterns like memory, compaction, and system-prompt injection. This is why skills-first is a constant refrain. A throwaway chat is a candle. A skill is a power outlet.

So what do you actually get, by the numbers? At version two-point-oh, the surface looks like this, and remember the counts drift weekly, so treat this as a snapshot. Sixty-four agents, specialized sub-agents for planning, review, build-fixing, and language-specific work. Two hundred and sixty-two skills, covering workflows and domain knowledge. Eighty-four commands, slash entries plus an opt-in archive of retired shims. Hooks, which are automations across the session lifecycle. Rules, covering common standards plus per-language packs for TypeScript, Python, Go, Swift, PHP, ArkTS, and more. MCP configs, which are connector definitions for external services. Cross-platform scripts for install, hooks, sessions, and orchestration. A Rust control-plane prototype that is the early version of ECC two-point-oh. And a desktop graphical interface to browse it all.

Don't be intimidated by those counts. You will use a handful of agents and a dozen skills ninety percent of the time. The rest is there when a niche need arises, whether that's Kotlin build errors, HarmonyOS apps, prediction-market research, or even video editing. Yes, really.

So, the key takeaways. ECC is Everything Claude Code, a harness performance system born from more than ten months of real daily use. A harness wraps a model and lets it act, and ECC tunes the harness, not the model. Five principles run through everything: agent-first, test-driven, security-first, immutability, and plan-before-execute. ECC bets on durable, compounding workflows over throwaway prompts. And the big counts are a buffet. You only plate what you need.
