第四章。核心概念：六大构建模块。

ECC 里面所有东西，都是由六种东西组成的。学会这六种，整个系统就会清晰起来。这一章是地图，而接下来的章节会逐个放大来看。

我们快速看一下这六样东西。第一，agents。这些是你委派工作给它的、有明确范围的专家。它们以单个文件的形式存在，而你互动的方式，就是由 orchestrator 委派工作给它们，不过你也可以直接要求其中一个。第二，skills，这些是主角。它们是可重用、多步骤、捆绑着知识的工作流程。每一个都住在自己的文件夹。你调用它们，它们会被自动建议，或者 agents 会重用它们。第三，commands。这些是触发一个工作流程的斜杠项目，而你互动的方式，就是打一个斜杠和 command 的名字。第四，hooks。这些是由事件触发的自动化，由一个配置文件对应到小小的脚本，而你互动的方式，就是你不用互动。它们会自动启动。第五，rules。这些是要永远遵守的指引，有通用的，也有各语言的，而它们每一次会话都会被加载进上下文。而第六，MCP 配置。这些是连接外部服务的连接器。你逐个项目启用它们，而模型会调用它们暴露出来的工具。在这六样之中，skills 是标准的界面。当你拿不定主意，就用 skills 的角度去想。

这里有一个好用的方法，把这六样东西记进脑。Rules 是宪法：被动、永远存在、约束着一切。Skills 是战术手册：耐用的“我们在这里是这样做某件事的”。Commands 是捷径：方便、用打字触发，而且越来越只是 skills 之上的薄薄包装。Agents 是团队：orchestrator 把工作交给的专家，它们通常会用到 skills。Hooks 是反射动作：它们在事件发生时启动，不用任何人开口。而 MCP 是电话线：模型怎样接触到 GitHub、一个数据库，或者一个浏览器。一个单一的功能请求可能会碰到全部六样。一个 command 启动一个 skill，那个 skill 委派给一个 agent，全部受 rules 约束，每次编辑都有 hooks 启动，还有一个 MCP 服务器去抓取数据。那种编排就是 ECC。

我们放大来看 skills 对 commands 对 hooks，这三个主动的界面。这三个是人们最容易混淆的，因为三个都可以运行一个工作流程。区别在于触发方式和耐用程度。一个 skill 是被调用的，由你或者自动触发，而且它是耐用的单位。新逻辑就应该放这里。一个 command 是你打出来的，而 ECC 保留 commands 是为了兼容，但它正在把它们的逻辑迁移进 skills。一个 hook 是由事件触发的，例如一次文件编辑或者一个会话开始，而你永远不会打它出来。这里有一条经验法则。问一下你自己：这样东西是不是应该在每次发生某件事的时候，无论如何都要运行？如果是，那么它就是一个 hook。或者问：这样东西是不是一个我选择去运行的工作流程？如果是，那么它就是一个 skill，再加一个可选的 command 捷径。

在六大模块之上，还有两个横跨各处的能力，把一切联系在一起。它们有自己的章节，因为它们就是让 ECC 感觉有生命、而不是静态的东西。第一个是记忆。Hooks 保存你的会话状态，并且在下一次重新加载它，这样助手就不用每天从零开始。第二个是持续学习。重复的模式变成 instincts，而一个 evolve command 会把它们聚集成全新的 skills。这个目录是由你自己的使用成长出来的。

还有一个值得知道的前言区惯例。Agents 和 skills 其实只是 Markdown 文件，在最顶有一个小小的结构化头部。这个头部告诉框架怎样对待这个文件。你会不断见到这个模式：一个名字、一段关于什么时候用它的描述，而 agents 还会有它们应该用的工具和模型，至于 skills 就会注明它们的来源。这个描述字段挑起了重任。它就是框架用来决定什么时候要把这个 agent 或者 skill 拿出来的依据。含糊的描述会被忽略。精准的就会被采用。

那么，重点精华。ECC 是六大模块：agents、skills、commands、hooks、rules 和 MCP 配置。Skills 是标准。Commands 越来越只是薄薄的捷径。Hooks 是自动的。Rules 约束、agents 执行、MCP 连接，而记忆加学习让它随时间进步。Agents 和 skills 其实只是 Markdown 加一个结构化头部，而那个描述就是驱动自动选取的东西。

现在我们从你委派工作给它的团队开始讲：agents。
