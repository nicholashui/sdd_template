第八章。掛鈎，Hooks。

我哋由 hooks 係乜開始講。一個 hook 係一個喺特定事件啟動嘅自動化：一次工具呼叫、一個會話開始、上下文就嚟被壓縮。同 skills 同 commands 唔同，你永遠唔會呼叫一個 hook。係框架觸發佢。Hooks 係 ECC 嘅反射動作：嗰啲應該每次都自動發生、唔使你記得去做嘅格式化、把守、秘密資料掃描，同記憶儲存。喺 ECC 入面，一個設定檔宣告事件同配對器，而每一個項目都會執行一個細細嘅 Node 腳本。佢哋用 Node 而唔係內嵌 shell，係為咗跨平台嘅可靠性，呢個係 1.8 版引入嘅一個刻意嘅強化。

而家我哋行一行事件生命週期。主要嘅事件用 Claude Code 嘅名，而其他框架就對應返去呢啲。有會話開始，喺一個新會話開始時啟動，而 ECC 用佢嚟載入之前嘅上下文、偵測你嘅套件管理器，同注入一份記憶摘要。有工具使用前，喺一個工具運行之前啟動，而 ECC 用佢嚟提醒你用終端機多工器、把守一次 git push、掃描秘密資料、保護設定檔，同埋喺第一次編輯之前強制做一次事實查核。有工具使用後，喺一個工具完成之後啟動，而 ECC 用佢嚟格式化程式碼、型別檢查、執行品質關卡、對多餘除錯日誌作出警告，同更新度量。有一個工具使用失敗事件，喺一個工具出錯時為診斷捕捉失敗嘅上下文。有壓縮前，喺上下文壓縮之前啟動，並且先將重要嘅狀態儲存去一個檔案。仲有停止，喺一個會話結束時啟動，寫低一份會話摘要、批次格式化同型別檢查，並且抽取學到嘅嘢。

有幾個有代表性嘅 ECC hooks，你會見到佢哋嘅名。有一個合併咗嘅 shell 預檢，捆綁咗品質、多工器、push 同關卡嘅檢查。有一個關卡守衛事實強制 hook，會阻止每個檔案嘅第一次編輯，並且要求調查，喺容許改動之前問：邊個引入緊呢個、個結構係乜，同用戶實際上問咗乜。有一個設定保護 hook，會阻止對 linter 同 formatter 設定嘅編輯，引導 agent 去修程式碼，而唔係削弱個設定。有一個品質關卡 hook，喺編輯之後運行。有一個 console 警告 hook，對付多餘嘅除錯日誌。有一個上下文監察 hook，會注入關於上下文耗盡、高成本、範圍蔓延，或者工具回圈嘅警告。有一個會話開始引導，負責記憶同套件管理器偵測。仲有壓縮同停止嘅 hooks，負責記憶留存。設定裏面真正嘅 command 字串睇落好嚇人，因為每一個都要喺好多可能嘅安裝位置之中，先引導到正確嘅外掛根目錄，然後先委派畀真正嘅腳本。你永遠唔使親手寫嗰啲。安裝器會生成佢。

呢度係 ECC 其中一個最好嘅功能：你可以唔使編輯任何檔案就調校 hooks。你用環境變數去控制 hook 嘅行為。有一個嚴格度設定檔，可以係 minimal、standard 或者 strict，預設係 standard。你可以用 hook 嘅 id，以逗號分隔嘅清單，去停用特定嘅 hooks。你可以為低上下文或者本機模型，封頂甚至關閉會話開始嘅上下文注入。你可以設定一個以日計嘅會話保留視窗。你可以保留上下文、範圍同回圈嘅警告，同時隱藏成本估算。而你可以選擇加入管治事件捕捉。每一個 hook 都標籤住佢喺邊啲設定檔下面運行，所以 minimal 設定檔會靜靜雞咁丟低啲比較嘈嘅，而 strict 就會將所有嘢開晒。

而家，一個關於 hooks 檔案重複陷阱嘅警告。呢個係最多人匯報嘅 hook 問題。Claude Code 2.1 版同之後，會按照慣例自動載入一個外掛嘅 hooks 設定。如果你又喺外掛清單裏面宣告 hooks，或者喺外掛安裝之後將 hooks 設定複製入你嘅設定檔，你就會得到一個 hooks 檔案重複嘅錯誤。呢度係點樣避免佢。貢獻者永遠唔應該喺外掛清單裏面加一個 hooks 欄位，而一個迴歸測試會強制執行呢點。外掛用戶唔應該將 hooks 複製入佢哋嘅設定檔，因為佢哋已經載入咗。而手動用戶應該透過安裝器、用 hooks-runtime 模組嚟安裝 hooks，咁就會將佢哋寫入啱嘅位置，並且唔郁個設定檔。

我哋睇吓一個 hook 嘅概念形狀。如果你剝走個引導包裝，一個 hook 項目其實只係幾個欄位。有一個配對器，講呢個適用於邊啲工具或者事件，例如一次 shell 呼叫、一次編輯，或者所有嘢。有個 command，係要執行嘅腳本。有非同步同逾時嘅設定，咁個 hook 就唔會阻塞個 agent，而佢嘅運行時間有封頂。仲有一個 id，係你停用 hooks 時用嘅把手。仲有一個退出碼慣例：淨係喺阻擋係刻意嗰陣先用一個失敗碼退出，否則就乾淨咁退出，而啲訊息應該永遠都可以採取行動。指南入面一個最小嘅手寫範例，係一個配對 JavaScript 或者 TypeScript 檔案編輯嘅 hook，grep 一句多餘嘅除錯日誌，然後印一個提醒叫你移除佢。

推薦用簡單嘅方法建立 hooks，因為親手寫 hook 設定好麻煩。短篇指南推薦用 hookify 嘅做法：執行 hookify command，用對話方式描述你想要乜，佢就會幫你生成個 hook。ECC 仲內附了幾個配套 commands，去列出、設定，同攞 hookify 嘅協助。如果你真係親手寫一個，就用提供嘅腳本去驗證佢。

咁實際上 hooks 畀到你乜呢？喺 standard 設定檔開住 hooks，一個普通嘅編輯會話會靜靜咁得到：編輯之後自動格式化同型別檢查，咁就唔會再唔記得執行 formatter；喺有風險嘅工具呼叫之前同提示提交之前偵測秘密資料；一道 git-push 審查關卡，咁啲 push 就唔會冇聲冇息；一個用終端機多工器去運行長時間 commands 嘅提醒；跨會話嘅記憶留存；餵畀 instincts 嘅持續學習捕捉；同一個喺你撞牆之前警告你嘅上下文監察。咁就係喺零持續努力嘅情況下，攞到好多資深工程師嘅紀律。

咁，重點精華。Hooks 係喺工具同會話事件上面嘅自動反射動作，將一個設定檔對應到 Node 腳本。生命週期由會話開始，去到工具使用前、工具使用後、壓縮前，再到停止。你喺運行時用 hook 設定檔同停用 hooks 嘅環境變數去調校佢哋，唔使編輯檔案。永遠唔好雙重載入 hooks，即係嗰個 hooks 檔案重複陷阱，並且用啱嘅方法安裝佢哋。而且用 hookify command 去用描述建立 hooks，而唔係親手寫設定。
